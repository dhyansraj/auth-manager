* Overview

The auth platform provides multi-tenant authentication-as-a-service for tenant
apps running on the beelink k3s cluster. Each tenant gets a Keycloak realm with
its own users, IdPs (Google/GitHub), client apps, custom theme, and SMTP
configuration. Operators (you) manage tenants through =auth-manager= via the
=admin-ui=; tenant teams integrate their apps using the published auth
libraries (=mcp-mesh-auth-lib= for Java/Python, =@mcpmesh/auth-lib-react= for
React).

The platform runs in two fully-isolated environments, side-by-side on the
same cluster: =prod= (HA, real users) and =dev= (single-replica, throwaway
tenants for testing). Both share the same chart + image registry; only sizing
and infra targets differ.

* Operator workflows

** Logging in (prod)

1. Browse to https://auth.mcp-mesh.io/admin/
2. Hit "Sign in" → redirects through BFF → KC realm =dev= login page
3. Enter platform-admin credentials
4. Land on Tenants list

The platform realm is =dev= (yes, even in prod — naming legacy, doesn't
imply "dev environment"). It holds platform operator users only; tenant
users live in their own =t-<slug>= realms.

** Logging in (dev)

Same flow at https://auth-dev.mcp-mesh.io/admin/.

Current dev admin credentials (rotate via UI when comfortable):
- username: =admin=
- email:    =admin@platform.dev=
- temporary password: (recorded separately when seeded; KC forces change on first login)

DEV badge in header confirms you're in the dev environment.

To distinguish the two in Chrome's password manager, save dev's credentials
under a DIFFERENT username (e.g., =dev-admin=) than prod's. Chrome groups
credentials by registered domain (mcp-mesh.io) and will conflate them
otherwise.

** Creating a new tenant

In admin UI → Tenants → "+ New tenant":

1. **Basics**: slug (e.g., =happyfeet=), display name, primary admin email,
   environment (DEV / PROD — affects only which env you're operating in;
   each env has its own list).
2. **Apps**: at minimum one CONFIDENTIAL_BACKEND client. Pick from
   profile dropdown (CONFIDENTIAL_BACKEND / SPA_PKCE / SERVICE_ACCOUNT_ONLY).
   Tick =USER_LIST= on service account permissions if app needs to call
   platform admin API for user lookups.
3. **Hostnames**: primary public hostname. If it's =*.mcp-mesh.io= we own
   the DNS; otherwise tenant needs to point CNAME at our CF tunnel.
4. **Identity Providers**: Google / GitHub toggles. Each must have the
   tenant's broker redirect URI added to its OAuth console (see
   =01-broker-urls.md= in the onboarding bundle).
5. **Email**: From address (defaults =noreply@mcp-mesh.io= until tenant
   authenticates own domain), display name (defaults to tenant displayName).
6. **Default roles**: pick from manifest if uploading one.
7. **Manifest** (optional): upload =tenant-manifest.yaml= to seed
   permissions + role bundles.
8. **Branding** (optional): upload theme zip.
9. **Review + Generate**: creates KC realm, clients, IdPs, SMTP config, CF DNS
   CNAME, tunnel ingress rule, theme upload. Returns onboarding bundle for
   the tenant team.

Tenant-detail tabs are deep-linkable via =?tab== and survive window refocus.

** Configuring Identity Providers post-creation

Tenant detail → Identity Providers tab. Three toggle cards:
- **Google** — enabled if =google= creds present in =platform-oauth-providers= Secret
- **GitHub** — same for GitHub
- **Username / Password** — UX-only toggle (realm attribute drives
  =mcp-no-password= body class on the theme; CSS hides the password form
  when set). Server-side login flow is unchanged; KC will still accept
  password POSTs if anyone bypasses the UI.

Server-side guard: refuses to disable last login method (zero IdPs + password
off would lock everyone out).

Per-tenant disabled IdPs are persisted in =tenant.settings.disabledIdps=
(JSONB). IdentityProvidersBootstrap respects this on startup so deploys
don't resurrect operator-removed providers.

** Configuring email

Tenant detail → Email tab.

Three fields per tenant:
- From address (must match an authenticated SendGrid domain OR fall back to
  =noreply@mcp-mesh.io=)
- From display name (defaults to tenant display name)
- Reply-to address (optional; blank = KC defaults to From)

**Domain authentication** (one-shot, automated when tenant CF zone is in our
account):
1. Enter the domain to authenticate (e.g., =niralishappyfeetindia.com=)
2. Click "Authenticate domain"
3. Backend: SendGrid POST =/v3/whitelabel/domains= → returns 3 CNAMEs
4. Backend: CF API creates each CNAME with =proxied: false= in the tenant's
   CF zone (looked up via =/zones?name=<parent>=). If zone not in our account
   → CNAMEs returned as manual-add instructions.
5. Backend: SendGrid POST =/validate=
6. Click "Re-verify" after DNS propagates (~30s)
7. From address can now use the authenticated domain

*Fixed (#104)*: the domain-auth =valid= flag now persists on validate/re-verify
— the service reads the authoritative SendGrid =/validate= POST result rather
than the lagging GET, so no manual =UPDATE= on =sendgrid_domain_valid= is
needed anymore.

** Email templates (per-tenant transactional email)

The platform sends branded transactional email on a tenant's behalf — from
templates the *tenant owns*, delivered via the tenant's own authenticated
SendGrid domain. Used today for the invitation email, but tenants can add their
own types (digest, payment reminders, promotions, …).

*** Managing templates  [operator, =EMAIL_EDIT=]

Tenant detail → Email tab → *Email templates*. Per type:
- *Download starter* — zip skeleton: =template.html= (logic-less *Mustache*),
  =subject.txt= (Mustache), =assets/= (inline images), plus a README listing
  the available variables.
- *+ New template* — pick a free-form =typeKey= (lowercase slug).
- *Upload / Replace* — drop the edited =.zip= (pre-scanned on upload).
- *Preview* — renders in an iframe (=[var]= placeholders, assets inlined; loop
  sections render exactly one =[placeholder]= row, inverted ={{^x}}= sections
  are suppressed).
- *Delete*.

Templating is logic-less *Mustache*: variables from the send model as
={{var}}=; sections/loops via ={{#list}}…{{/list}}= and ={{^x}}…{{/x}}=; inline
images via ={{asset:<name>}}= (a file in =assets/=, embedded as a CID part).
*Full email HTML is allowed* — tables, presentational attributes, inline CSS,
bulletproof table buttons — while =<script>=, =on*= handlers,
=<iframe>/<object>/<form>=, inline =<svg>=, and =javascript:= links are
stripped; =href= is restricted to =http=/=https=/=mailto=/=tel=. (Email uses a
dedicated *permissive* sanitizer, NOT the tighter login-theme one.)

*** Reserved =invitation= type

Upload a template with =typeKey=invitation= to override the default invite
email (the branded "sign in to get started" mail). The invite flow uses the
tenant's =invitation= template if present, else the platform default.
Variables: ={{tenantDisplayName}}=, ={{inviteUrl}}= (or ={{ctaUrl}}=, same
value), ={{recipientEmail}}=, ={{inviterName}}= (the inviting operator's
display name — sent by the admin UI; optional in the API).

*** App-send API  [tenant backend, =EMAIL_SEND=]

=POST {AUTH_MGR_BASE}/api/v1/tenants/{tenantUuid}/emails/{typeKey}= with body
={ "to": "...", "model": { … } }=, gated =EMAIL_SEND= (grant on the backend
service account via Apps → Service Account Permissions). The IAM renders the
tenant's =typeKey= template with =model= and sends via SendGrid using the
tenant's From/DKIM. Returns =202 {"sent":true}=. The reserved =invitation= key
returns =409= — use the user-invite endpoint instead (which also provisions the
account).

=AUTH_MGR_BASE=: dev =https://auth-dev.mcp-mesh.io=, prod
=https://auth.mcp-mesh.io= (endpoints under =/api/v1/...=).

*** Rate limits

Per-tenant Redis counters protect domain reputation: defaults =100/min= +
=5000/day= (=auth-manager.email.rate-limit.{per-minute,per-day,enabled}=,
env-overridable). Over the limit → =429= + =Retry-After=. *Fail-open* if Redis
is unreachable. Per-tenant overrides:
=GET/PUT ${AUTH_MGR_BASE}/api/v1/tenants/{tenantUuid}/email/rate-limit=
(=TENANT_VIEW= to read, =TENANT_EDIT= to write; =null= clears an override back
to the platform default). Admin UI: Email tab → *Send rate limits* card.

** Theming

**Stock theme**: tenants inherit =mcpmesh-flexible= parent theme. Provides
4 layouts (centered / split-left / split-right / bleed) selected via
=mcpLayoutVariant= message in tenant's =messages_en.properties=.

**Per-tenant override**: tenant detail → Branding tab → "Download starter"
gets the zip skeleton. Edit, re-zip, upload. ConfigMap regenerated,
=ThemeApplier= patches KC StatefulSet =restartedAt= annotation to trigger
rolling restart (~3-5 min). Theme is live cluster-wide.

** Service account permissions

For tenant backends needing to call platform admin API (e.g., list users
by role for dropdowns):

Tenant detail → Apps tab → click the backend client row → modal opens with
checkboxes. Tick =USER_LIST= (or other perms from preset
=[USER_LIST, USER_INVITE, USER_DISABLE, AUDIT_VIEW]=). Save. Audited.

Two email perms also live here:
- =EMAIL_SEND= — call the transactional email *send* API. This is the perm a
  tenant *backend* grants to its service account so it can send mail via the
  platform (see *Email templates → App-send API* above).
- =EMAIL_EDIT= — manage the tenant's email *templates* (download starter,
  upload/replace, delete). Operator-side.

Tenant backend then mints a =client_credentials= token using its
=AUTH_LIB_CLIENT_ID= + =AUTH_LIB_CLIENT_SECRET= and calls
=GET ${AUTH_MGR_BASE}/api/v1/tenants/<slug>/users?role=<role>= with the
bearer token.

** Login methods

Identity Providers tab also has a **Username / Password** toggle.

When toggled off:
- Realm attribute =mcpmesh.passwordLoginEnabled = "false"= written
- =mcpmesh-flexible= template reads attribute, adds =mcp-no-password=
  body class
- CSS hides username/password form, sign-in button, forgot-password link
  (scoped to the login page — no longer hides the Submit button on the
  Review Profile / update-profile page; fixed)
- Only IdP buttons render

Server-side login flow unchanged (this is UX gating, not enforcement). A
determined attacker could POST credentials directly to KC's
=/login-actions/authenticate= — server-side enforcement requires a custom
KC SPI, out of scope today.

** Invite-only access (per-tenant)

Opt-in per tenant. When ON, only users an operator has provisioned/invited
may sign in via a brokered IdP (Google etc.) — unknown emails are *rejected
at Keycloak*, and self-registration is disabled. OFF by default, so
self-signup tenants are unaffected. Unlike the UX-only "Username / Password"
toggle above, this is real server-side enforcement.

Enable: Tenant detail → Identity Providers tab → *Invite-only* toggle.
API: =PUT ${AUTH_MGR_BASE}/api/v1/tenants/<slug>/identity-providers/registration=
body ={"inviteOnly": true}= (gated =IDP_EDIT=); =GET= the same path returns
={inviteOnly, registrationAllowed}=. Persisted in =tenant.settings.inviteOnly=
(JSONB); the startup reconcile re-applies it per realm on every deploy.

How it works — the per-realm =first broker login auto-link= flow:
- ON: =idp-detect-existing-broker-user= REQUIRED (detects the pre-provisioned
  account by email), =idp-create-user-if-unique= DISABLED (no auto-create),
  =Handle Existing Account= subflow REQUIRED (so =idp-auto-link= runs and links
  Google to the invited account), realm =registrationAllowed=false=.
- OFF: detect DISABLED, create-if-unique ALTERNATIVE, Handle Existing Account
  ALTERNATIVE, =registrationAllowed=true=.
Invited user → linked + signed in; uninvited → denied, no account created.
Pairs with #94 auto-link.

*Gotcha for future flow edits*: KC ignores ALTERNATIVE executions sitting at
the same level as a REQUIRED one — that is why Handle Existing Account must be
flipped to REQUIRED in invite-only mode, else =idp-auto-link= is silently
skipped and every login fails with =invalid_user_credentials=.

*** Inviting users

Create the invited (provisioned) account two ways:
- Admin UI: Tenant detail → Users tab → *+ Invite user* (email + roles). The
  Users tab also has a per-row *mark verified* action for unverified users.
- API (for tenant apps): =POST ${AUTH_MGR_BASE}/api/v1/tenants/<tenantUuid>/users=
  body ={"email","firstName","lastName","roles":["user-viewer"],"sendInvite":true}=
  (gated =USER_INVITE=). The tenant backend calls this with its service-account
  =client_credentials= token (grant =USER_INVITE= via Apps → service-account
  permissions). It pre-creates the KC user (so invite-only auto-link lets them
  in) and emails the invite.

Invites to brokered / Google-only tenants now send a *branded "sign in to get
started" email* — sent by auth-manager via SendGrid, not the old KC
=UPDATE_PASSWORD= mail. It's overridable per tenant via the reserved
=invitation= email template (see *Email templates* above).

*App-vs-IAM division*: the IAM owns the user store + invite email + invite-only
enforcement; the app owns invitation *policy* (who may invite, per-user quota,
referral tracking). On "invite a friend" the app checks its quota, then calls
the IAM invite API — the IAM provisions the user and emails them. Apps should
NOT keep their own allow-list: the IAM user list *is* the allow-list.

Caveats:
- Invite endpoint is UUID-keyed (=/tenants/{tenantUuid}/users=) — hand the app
  its tenant UUID (slug alias TBD).
- The invite email currently uses KC =UPDATE_PASSWORD= action, which is wrong
  for a Google-only tenant (prompts for a pointless password). Switch to a
  "sign in with Google" invite before apps rely on it (follow-up).
- The service-account client must NOT use lightweight access tokens, or the
  =USER_INVITE= role claim is stripped and the call 403s.

** Deleting a tenant

Tenant detail → "Delete" button (top right).
- Soft-deletes the DB row (=tenants.deleted_at= set)
- Disables the KC realm (=enabled = false=)

** Three roles

| Role         | Who                                     | Does                                                              |
|--------------+-----------------------------------------+------------------------------------------------------------------|
| *User*       | Operator (Dhyan)                        | All admin-UI actions, Google console, secret hand-off, deploys   |
| *IAM Claude* | auth-platform side                      | Produces the spec (client/manifest/routes/env), verifies platform |

** The integration contract

The tenant owns ZERO auth code. Its footprint:
- A UI server (nginx serving a static SPA, or node).
- A backend/api-gateway that ONLY validates the BFF-injected =Authorization:
  Bearer <jwt>= via the published auth-lib (=mcp-mesh-auth-lib= Java/Python) and
  may inject =X-User-Email=/=X-Kc-Sub= to internal services (see happy-feet
  =AuthFilter=).
- Internal services/agents — no auth; they trust the gateway-injected identity.

The platform owns everything else:
- Login / session / refresh / logout — platform-edge *BFF* (=/_bff/login=,
  =/_bff/me=, =/_bff/logout=). Frontend uses =@mcpmesh/auth-lib-react=
  (=PlatformAuthProvider= + =bffFetch=); on 401 it redirects to =/_bff/login=.
- Identity providers (Google, GitHub, …) — per-realm KC brokering. The tenant
  configures NOTHING for Google; one shared platform Google app serves all
  realms.
- Invitations + user management — auth-manager API / admin-UI (=USER_INVITE=
  etc.). Access is gated by token permissions, NOT an app-side allowlist.

** Dev vs prod model

A tenant's *dev* runs in the *isolated dev environment* (=auth-platform-dev=) —
its own Keycloak, DB, edge, Cloudflare tunnel, and Google client, separated from
prod. The app runs *locally* on a dev machine; the dev edge proxies to it over
the LAN. (See =Dev environment architecture= below.) Prod is a separate tenant
in the prod control plane.

| Aspect          | Dev tenant                                                   | Prod tenant                                     |
|-----------------+--------------------------------------------------------------+-------------------------------------------------|
| Control plane   | =auth-dev.mcp-mesh.io/admin= (=auth-platform-dev=)           | =auth.mcp-mesh.io/admin= (=auth-platform=)      |
| Realm / KC      | =t-<slug>-dev= on dev KC =auth-dev.mcp-mesh.io=              | =t-<slug>= on prod KC =auth.mcp-mesh.io=        |
| Public hostname | =<slug>-dev.mcp-mesh.io= (dev tunnel)                        | =<slug>.<domain>= (prod tunnel)                 |
| Issuer          | =https://auth-dev.mcp-mesh.io/auth/realms/t-<slug>-dev=      | =https://auth.mcp-mesh.io/auth/realms/t-<slug>= |
| Route targets   | dev machine over LAN (=http://192.168.10.50:3000= / =:8080=) | in-cluster svc DNS                              |
| App runs        | LOCALLY on the dev machine                                   | deployed to k3s                                 |
| Google client   | dev Google client (isolated)                                | prod Google client                              |
| Deploy needed   | NO                                                           | YES                                             |

A lighter alternative — a =-dev= tenant in the *prod* control plane (e.g.
=safesound-dev=) — shares the prod KC + Google client + tunnel: simpler, but not
isolated. Prefer the isolated dev env for new tenants.

** Step-by-step

*** Phase 1 — Discovery & spec  [IAM Claude + App Claude]
- *App Claude*: describe the stack (backend language, frontend, what the
  services/"agents" are) and the role/permission model needed.
- *IAM Claude*: turn that into the concrete spec — the client, the
  =tenant-manifest.yaml= (permissions + roles), the route rules, the dev env
  block — and verify anything non-obvious against the platform code.
- ⚠️ *Roles-as-permissions*: a composite role NAME never appears in the JWT —
  =resource_access.<client>.roles= carries the atomic PERMISSION (client-role)
  strings only. Model the things the app checks (e.g. =ADMIN=) as
  *permissions*, bundled into roles:
  #+BEGIN_SRC yaml
  permissions:
    - id: USER
      description: Authenticated user
      client: backend
    - id: ADMIN
      description: Administrator
      client: backend
  roles:
    - name: user
      permissions: [USER]
    - name: admin
      permissions: [USER, ADMIN]
  defaultRoles: [user]
  #+END_SRC
  Token then carries =["USER"]= for a user and =["USER","ADMIN"]= for an admin;
  the app checks for =ADMIN=.

*** Phase 2 — Provision the tenant in the IAM  [User, admin UI]
All in =https://auth.mcp-mesh.io/admin/=:
1. *Tenants → New tenant*: slug =<slug>-dev= (dev) or =<slug>= (prod), display
   name, admin email.
2. *Apps*: add one app, profile =CONFIDENTIAL_BACKEND=, client id =backend=
   (the resource-server client the gateway validates against; it needs NO
   redirect URI — the BFF login uses the per-realm =usermanagement= client).
3. *Routes*: targets + rules.
   - Dev targets: =frontend → http://192.168.10.50:<ui-port>=,
     =backend → http://192.168.10.50:<gw-port>= (dev machine LAN IP).
   - Prod targets: in-cluster svc DNS.
   - Rules (default): =/api → REQUIRED → backend=, =/* → OPTIONAL → frontend=.
     Gate a path with =requiredPermission=; use =bypassCsrf= / =maxBodyMb= /
     =stripPrefix= as needed.
4. *Hostnames*: =<slug>-dev.mcp-mesh.io= (dev) / =<slug>.<domain>= (prod). The
   platform auto-creates the CF CNAME; an =*.mcp-mesh.io= host routes via the
   wildcard ingress to the edge with no tunnel edit (a custom apex auto-adds an
   ingress entry).
5. *Permissions / Roles*: upload the =tenant-manifest.yaml= from Phase 1.
6. *Client secret*: grab =backend='s secret from the create result / onboarding
   bundle (or KC console → realm → Clients → backend → Credentials). Hand
   =client_id= + =client_secret= to the tenant *out of band* (not over the
   relay).
7. *Users*: create test users; assign one the =admin= role. The wizard
   provisions the tenant admin user with =emailVerified=true=; mark other
   users verified via the Users-tab *mark verified* action or
   =POST ${AUTH_MGR_BASE}/api/v1/tenants/{tenantUuid}/users/{userId}/verify-email=
   (gated =USER_INVITE=).

*** Phase 3 — App integration  [App Claude]
- Delete any tenant-side OAuth/OIDC/session code (e.g. an nginx Lua Google flow).
- Frontend: wrap in =@mcpmesh/auth-lib-react= =PlatformAuthProvider=; use
  =bffFetch= for API calls; on 401 redirect to =/_bff/login= (see happy-feet
  =AuthContext.tsx= / =bffAuth.ts=).
- Backend: add =mcp-mesh-auth-lib=; validate the BFF Bearer against the realm
  issuer; implement =/api/me= via =MeResponseBuilder.fromJwt= (Java) /
  =build_me_response= (Python); inject =X-User-Email= to internal services if
  needed (happy-feet =AuthFilter=). Replace any =is_admin= subrequest with a
  token-permission check (=ADMIN=).
- Env (dev):
  #+BEGIN_SRC sh
  AUTH_LIB_ISSUER_URI=https://auth.mcp-mesh.io/auth/realms/t-<slug>-dev
  AUTH_LIB_CLIENT_ID=backend
  AUTH_LIB_AUDIENCES=backend
  AUTH_LIB_CLIENT_SECRET=<from User, out of band>
  AUTH_LIB_PERMISSIONS_SOURCE=claims
  # server-to-server, only if the app calls the auth-manager API:
  KC_BASE=https://auth.mcp-mesh.io/auth
  AUTH_MGR_BASE=https://auth.mcp-mesh.io
  #+END_SRC
  Prod flip: =iss= host stays =auth.mcp-mesh.io=; realm → =t-<slug>=; swap the
  secret; set =AUTH_MGR_BASE= to the in-cluster svc URL.

*** Phase 4 — Validate end-to-end  [App Claude + User]
- Run the app locally; bind the UI and gateway to =0.0.0.0= so the cluster edge

- Browse to =https://<slug>-dev.mcp-mesh.io= → BFF redirects to login → sign in

  =https://auth-dev.mcp-mesh.io/auth/realms/t-<slug>-dev/broker/google/endpoint=

*** Phase 6 — Prod cutover  [User + App Claude]

* Dev environment architecture

Tenant *dev* runs in the *isolated dev environment* — the =auth-platform-dev=
namespace, fully separated from prod: its own Keycloak, Postgres, Redis, edge,
Cloudflare tunnel, and Google OAuth client. The tenant's app runs *locally* on a
developer machine; the dev edge proxies to it over the LAN.

** Components

| Piece            | Value                                                                                        |
|------------------+----------------------------------------------------------------------------------------------|
| Namespace        | =auth-platform-dev=                                                                           |
| Admin UI         | =https://auth-dev.mcp-mesh.io/admin/= (operator =dev-admin=, platform realm =dev=)            |
| KC (issuer host) | =auth-dev.mcp-mesh.io= (=platform-kc-dev=, =KC_HOSTNAME_STRICT=true=)                         |

| Google client    | "MCP Mesh Auth Service - Dev" =943290783081-bgih…= (creds in dev =platform-oauth-providers=) |

The dev =cloudflare-api= secret's =tunnel-id= points at the dev tunnel, so the
dev auth-manager (reads =CF_TUNNEL_ID=) auto-creates each dev tenant's DNS CNAME
→ dev tunnel.

** Hostname + TLS scheme (important)

- Dev tenant hostname = single-level =<slug>-dev.mcp-mesh.io=.
- ⚠️ Do NOT use a two-level wildcard like =*.dev.mcp-mesh.io=. Cloudflare free
  Universal SSL covers only the apex + a *single-level* =*.mcp-mesh.io=, so a
  two-level name fails the TLS handshake (=sslv3 alert handshake failure=); it
  would require paid Advanced Certificate Manager.
- Routing: the tenant's specific CNAME =<slug>-dev.mcp-mesh.io → dev tunnel=
  overrides the prod =*.mcp-mesh.io= wildcard (a specific DNS record wins), so it
  reaches the dev tunnel → dev edge. TLS is covered (single-level). Auto-created
  by the dev auth-manager.

** Request path

#+BEGIN_SRC text
  browser → https://<slug>-dev.mcp-mesh.io
          → (specific CNAME) dev CF tunnel
          → dev edge (catch-all ingress)
          → host→tenant routing (dev Redis)
          → BFF login at dev KC (auth-dev.mcp-mesh.io) + proxy /api,/ to the
            Route targets (dev machine LAN IP, e.g. 192.168.10.50:3000 / :8080)
#+END_SRC

Fix:
#+BEGIN_SRC bash

#+END_SRC

** CNPG superuser pinning — one-time migration (#92)

The code/manifest fix prevents drift going forward, but the EXISTING prod and
dev clusters were created WITHOUT =enableSuperuserAccess= and with a
CNPG-auto-generated superuser secret. They each need a one-time migration to
adopt the pinned-secret model. Until that's done, they will still drift and the
LEGACY recovery steps above remain necessary.

Per cluster (prod shown; for dev use the =-dev= names + =auth-platform-dev= NS):

1. Create/replace the superuser secret as a =kubernetes.io/basic-auth= secret
   with =username=postgres= and the password the DB CURRENTLY uses (read it
   from the existing secret first so we don't change the live password):
   #+BEGIN_SRC bash
   PASS=$(ssh beelink1 'kubectl get secret platform-pg-superuser-secret -n auth-platform -o jsonpath="{.data.password}" | base64 -d')
   ssh beelink1 "kubectl create secret generic platform-pg-superuser-secret \
     -n auth-platform --type=kubernetes.io/basic-auth \
     --from-literal=username=postgres --from-literal=password='${PASS}' \
     --dry-run=client -o yaml | kubectl apply -f -"
   #+END_SRC
2. Ensure the CR has =enableSuperuserAccess: true=. For prod, re-apply the
   updated manifest:
   #+BEGIN_SRC bash
   ssh beelink1 'cd ~/workspace/github/auth-manager && kubectl apply -f deploy/cnpg/pg-cluster.yaml'
   #+END_SRC
   For dev, re-running =scripts/deploy-beelink-dev.sh= applies the inline CR
   with the flag set (or =kubectl patch cluster platform-pg-dev -n
   auth-platform-dev --type=merge -p '{"spec":{"enableSuperuserAccess":true}}'=).
3. Let CNPG reconcile — it syncs the =postgres= role password to the secret.
   Verify =psql -U postgres -c 'SELECT 1'= works with the secret's password.

After this, drift stops and the LEGACY =ALTER USER= pre-steps + recovery runbook
are no longer needed for that cluster.

** KC realm session/token defaults

New tenant realms are created with =accessTokenLifespan=900= (15 min),
=ssoSessionIdleTimeout=14400= (4 h), =ssoSessionMaxLifespan=86400= (24 h).
The manual bump via KC admin console (realm → Sessions/Tokens) is only needed
for realms created before =auth-manager 0.1.80=.

* Known gotchas / open issues

** #92 CNPG superuser drift on every redeploy (FIXED IN CODE — migration pending)
The =postgres= password used to be reset by CNPG on every reconcile because
=enableSuperuserAccess= defaulted to false (CNPG ≥1.21). Fixed in code via
=enableSuperuserAccess: true= + a pinned operator-owned basic-auth superuser
secret (=username=postgres=) — CNPG now syncs the role password to the secret
instead of rotating it. A ONE-TIME live-cluster migration is still required for
the existing prod + dev clusters (see "CNPG superuser pinning — one-time
migration" under Common ops). The LEGACY =ALTER USER= workaround remains only
until each cluster is migrated.

** #115 cf-setup-tunnel.sh wipes tenant ingress rules (CRITICAL)
Re-running the script destroys CloudflareTunnelService-added tenant rules
(=niralishappyfeetindia.com=, =safeandsoundhouses.com=, =maya-ai.ink=). Must
manually restore using the curl block in the Deployment section. Real fix:
read current ingress, MERGE static + tenant rules, preserve unknowns.

** #104 sendgrid_domain_valid doesn't persist after revalidate — RESOLVED
Fixed: the =valid= flag now persists on validate/re-verify (service reads the
authoritative SendGrid =/validate= POST result, not the lagging GET). No manual
=UPDATE= needed.

** #103 PermissionsTab error display
Operators get raw =ApiError: HTTP 400 {timestamp...}= JSON dumped on upload
failure. Should parse structured error code and show a friendly message.

** #113 deploy-beelink-dev.sh polish
First-install bugs: doesn't enable CNPG superuserAccess automatically, doesn't
seed =postgres-password= / =keycloak-admin-password= into the chart's secret.

** #114 theme-dev-sync.sh missing — RESOLVED (2026-06-10)
Fixed: =scripts/theme-dev-sync.sh= exists (fswatch + kubectl cp into the dev
KC pod; see "Theme iteration in dev" under Common ops).

** #90 Wizard auto-verify admin email — RESOLVED
Fixed: the wizard provisions the tenant admin user with =emailVerified=true=;
other users via the Users-tab *mark verified* action or the verify-email API.

** #94 IdP auto-link operator user
When operator-provisioned user later logs in via Google, KC creates a NEW
account instead of linking to the existing one. Manual fix per user today.
