* Overview

The auth platform provides multi-tenant authentication-as-a-service for tenant
apps running on the beelink k3s cluster. Each tenant gets a Keycloak realm with
its own users, IdPs (Google/GitHub), client apps, custom theme, and SMTP
configuration. Operators (you) manage tenants through =auth-manager= via the
=admin-ui=; tenant teams integrate their apps using the published auth
libraries (=mcp-mesh-auth-lib= for Java/Python, =@mcpmesh/auth-lib-react= for
React).

The platform runs in two fully-isolated environments, side-by-side on the
same cluster: =prod= (HA, real users) and =dev= (single-replica, throwaway
tenants for testing). Both share the same chart + image registry; only sizing
and infra targets differ.

** Two-environment model

| Component        | Prod (=auth-platform=)    | Dev (=auth-platform-dev=)     |
|------------------+---------------------------+--------------------------------|
| CNPG cluster     | =platform-pg= (3 inst HA) | =platform-pg-dev= (1 inst)    |
| KC StatefulSet   | =platform-kc-keycloak= (3) | =platform-kc-dev-keycloak= (1) |
| Redis            | =platform-redis= (Sentinel) | =platform-redis-dev= (standalone) |
| auth-manager     | 2 replicas                 | 1 replica                       |
| admin-ui         | 2 replicas                 | 1 replica                       |
| platform-edge    | 3 replicas                 | 1 replica                       |
| smtp-relay       | 1 replica                  | (disabled — shares prod's)     |
| cloudflared      | 2 replicas                 | (disabled — uses prod's tunnel) |
| Public hostname  | =auth.mcp-mesh.io=         | =auth-dev.mcp-mesh.io=          |
| Theme cache      | enabled (prod perf)        | DISABLED (fast iteration)       |
| Platform realm   | =dev= (in prod KC)         | =dev= (in dev KC)               |
| Tenant realms    | =t-<slug>=                 | =t-<slug>=                      |

The dev env is a side-by-side replica of prod with separate state. Tenants
created in dev exist only in dev's KC + DB. Promotion is currently manual
(re-create in prod); future Phase 2 will add a migration tool (task #110).

* Operator workflows

** Logging in (prod)

1. Browse to https://auth.mcp-mesh.io/admin/
2. Hit "Sign in" → redirects through BFF → KC realm =dev= login page
3. Enter platform-admin credentials
4. Land on Tenants list

The platform realm is =dev= (yes, even in prod — naming legacy, doesn't
imply "dev environment"). It holds platform operator users only; tenant
users live in their own =t-<slug>= realms.

** Logging in (dev)

Same flow at https://auth-dev.mcp-mesh.io/admin/.

Current dev admin credentials (rotate via UI when comfortable):
- username: =admin=
- email:    =admin@platform.dev=
- temporary password: (recorded separately when seeded; KC forces change on first login)

DEV badge in header confirms you're in the dev environment.

To distinguish the two in Chrome's password manager, save dev's credentials
under a DIFFERENT username (e.g., =dev-admin=) than prod's. Chrome groups
credentials by registered domain (mcp-mesh.io) and will conflate them
otherwise.

** Creating a new tenant

In admin UI → Tenants → "+ New tenant":

1. **Basics**: slug (e.g., =happyfeet=), display name, primary admin email,
   environment (DEV / PROD — affects only which env you're operating in;
   each env has its own list).
2. **Apps**: at minimum one CONFIDENTIAL_BACKEND client. Pick from
   profile dropdown (CONFIDENTIAL_BACKEND / SPA_PKCE / SERVICE_ACCOUNT_ONLY).
   Tick =USER_LIST= on service account permissions if app needs to call
   platform admin API for user lookups.
3. **Hostnames**: primary public hostname. If it's =*.mcp-mesh.io= we own
   the DNS; otherwise tenant needs to point CNAME at our CF tunnel.
4. **Identity Providers**: Google / GitHub toggles. Each must have the
   tenant's broker redirect URI added to its OAuth console (see
   =01-broker-urls.md= in the onboarding bundle).
5. **Email**: From address (defaults =noreply@mcp-mesh.io= until tenant
   authenticates own domain), display name (defaults to tenant displayName).
6. **Default roles**: pick from manifest if uploading one.
7. **Manifest** (optional): upload =tenant-manifest.yaml= to seed
   permissions + role bundles.
8. **Branding** (optional): upload theme zip.
9. **Review + Generate**: creates KC realm, clients, IdPs, SMTP config, CF DNS
   CNAME, tunnel ingress rule, theme upload. Returns onboarding bundle for
   the tenant team.

** Configuring Identity Providers post-creation

Tenant detail → Identity Providers tab. Three toggle cards:
- **Google** — enabled if =google= creds present in =platform-oauth-providers= Secret
- **GitHub** — same for GitHub
- **Username / Password** — UX-only toggle (realm attribute drives
  =mcp-no-password= body class on the theme; CSS hides the password form
  when set). Server-side login flow is unchanged; KC will still accept
  password POSTs if anyone bypasses the UI.

Server-side guard: refuses to disable last login method (zero IdPs + password
off would lock everyone out).

Per-tenant disabled IdPs are persisted in =tenant.settings.disabledIdps=
(JSONB). IdentityProvidersBootstrap respects this on startup so deploys
don't resurrect operator-removed providers.

** Configuring email

Tenant detail → Email tab.

Three fields per tenant:
- From address (must match an authenticated SendGrid domain OR fall back to
  =noreply@mcp-mesh.io=)
- From display name (defaults to tenant display name)
- Reply-to address (optional; blank = KC defaults to From)

**Domain authentication** (one-shot, automated when tenant CF zone is in our
account):
1. Enter the domain to authenticate (e.g., =niralishappyfeetindia.com=)
2. Click "Authenticate domain"
3. Backend: SendGrid POST =/v3/whitelabel/domains= → returns 3 CNAMEs
4. Backend: CF API creates each CNAME with =proxied: false= in the tenant's
   CF zone (looked up via =/zones?name=<parent>=). If zone not in our account
   → CNAMEs returned as manual-add instructions.
5. Backend: SendGrid POST =/validate=
6. Click "Re-verify" after DNS propagates (~30s)
7. From address can now use the authenticated domain

**Known UX bug**: =sendgrid_domain_valid= flag doesn't reliably persist after
revalidate. Manual workaround: =UPDATE auth_manager.tenants SET
sendgrid_domain_valid = true WHERE slug='<slug>';= (task #104).

** Theming

**Stock theme**: tenants inherit =mcpmesh-flexible= parent theme. Provides
4 layouts (centered / split-left / split-right / bleed) selected via
=mcpLayoutVariant= message in tenant's =messages_en.properties=.

**Per-tenant override**: tenant detail → Branding tab → "Download starter"
gets the zip skeleton. Edit, re-zip, upload. ConfigMap regenerated,
=ThemeApplier= patches KC StatefulSet =restartedAt= annotation to trigger
rolling restart (~3-5 min). Theme is live cluster-wide.

** Service account permissions

For tenant backends needing to call platform admin API (e.g., list users
by role for dropdowns):

Tenant detail → Apps tab → click the backend client row → modal opens with
checkboxes. Tick =USER_LIST= (or other perms from preset
=[USER_LIST, USER_INVITE, USER_DISABLE, AUDIT_VIEW]=). Save. Audited.

Tenant backend then mints a =client_credentials= token using its
=AUTH_LIB_CLIENT_ID= + =AUTH_LIB_CLIENT_SECRET= and calls
=GET ${AUTH_MGR_BASE}/api/v1/tenants/<slug>/users?role=<role>= with the
bearer token.

** Login methods

Identity Providers tab also has a **Username / Password** toggle.

When toggled off:
- Realm attribute =mcpmesh.passwordLoginEnabled = "false"= written
- =mcpmesh-flexible= template reads attribute, adds =mcp-no-password=
  body class
- CSS hides username/password form, sign-in button, forgot-password link
- Only IdP buttons render

Server-side login flow unchanged (this is UX gating, not enforcement). A
determined attacker could POST credentials directly to KC's
=/login-actions/authenticate= — server-side enforcement requires a custom
KC SPI, out of scope today.

** Deleting a tenant

Tenant detail → "Delete" button (top right).
- Soft-deletes the DB row (=tenants.deleted_at= set)
- Disables the KC realm (=enabled = false=)
- Removes hostname routing (Redis route blob cleared)
- Removes CF tunnel ingress rule + DNS CNAME via CloudflareTunnelService

KC realm is NOT hard-deleted (data preserved for forensics). Operator can
purge manually via KC admin console if needed.

* Common ops

** Password rotation

*** SendGrid API key
1. SendGrid UI → Settings → API Keys → revoke old, create new (Mail Send
   scope only)
2. =kubectl create secret generic sendgrid-api -n auth-platform
   --from-literal=api-key=<NEW> --dry-run=client -o yaml | kubectl apply -f -=
3. =kubectl rollout restart deployment auth-platform-smtp-relay -n auth-platform=
4. Repeat for dev: same secret name in =auth-platform-dev= namespace

*** CF API token
1. Cloudflare dashboard → My Profile → API Tokens → rotate
2. Patch the =cloudflare-api= Secret with new value
3. Restart auth-manager + cloudflared (consumers)

*** KC admin password (prod)
1. Log into KC admin console
2. Master realm → Users → admin → Credentials → Reset
3. Update =auth-platform-auth-manager= Secret's =keycloak-admin-password= field
4. Restart auth-manager

*** CNPG superuser password (prod or dev)
The =postgres= password is now pinned to =platform-pg-superuser-secret= (prod)
/ =platform-pg-dev-superuser-secret= (dev) via =enableSuperuserAccess: true=
(#92). To rotate: update the secret's =password= (basic-auth, =username=postgres=)
and let CNPG reconcile it onto the role.

LEGACY (only until the live cluster is migrated — see "One-time migration"):
CNPG used to drift the password on every redeploy, requiring a manual
=kubectl exec -n <ns> <pg-pod-0> -c postgres -- psql -U postgres -c "ALTER
USER postgres WITH PASSWORD '<value from secret>';"= before each deploy.

*** Tenant client_secret (e.g., =happyfeet-backend=)
Admin UI → Apps → click app → "Regenerate secret" (or via KC admin console
→ realm → Clients → tenant client → Credentials → Regenerate)

** CNPG superuser drift recovery (LEGACY — until live clusters migrated)

NOTE: drift is fixed in code (#92) via =enableSuperuserAccess: true= + a pinned
operator-owned superuser secret. This runbook is only needed on clusters that
have NOT yet had the one-time migration applied (see "One-time migration"
below). Once migrated, the symptom below stops occurring.

Symptom: =FATAL: password authentication failed for user "postgres"= in
auth-manager logs. Data Services tab stuck on "Loading...".

Fix:
#+BEGIN_SRC bash
PASS=$(ssh beelink1 'kubectl get secret platform-pg-superuser-secret -n auth-platform -o jsonpath="{.data.password}" | base64 -d')
ssh beelink1 "kubectl exec -n auth-platform platform-pg-1 -c postgres -- psql -U postgres -c \"ALTER USER postgres WITH PASSWORD '${PASS}';\""

** CNPG superuser pinning — one-time migration (#92)

The code/manifest fix prevents drift going forward, but the EXISTING prod and
dev clusters were created WITHOUT =enableSuperuserAccess= and with a
CNPG-auto-generated superuser secret. They each need a one-time migration to
adopt the pinned-secret model. Until that's done, they will still drift and the
LEGACY recovery steps above remain necessary.

Per cluster (prod shown; for dev use the =-dev= names + =auth-platform-dev= NS):

1. Create/replace the superuser secret as a =kubernetes.io/basic-auth= secret
   with =username=postgres= and the password the DB CURRENTLY uses (read it
   from the existing secret first so we don't change the live password):
   #+BEGIN_SRC bash
   PASS=$(ssh beelink1 'kubectl get secret platform-pg-superuser-secret -n auth-platform -o jsonpath="{.data.password}" | base64 -d')
   ssh beelink1 "kubectl create secret generic platform-pg-superuser-secret \
     -n auth-platform --type=kubernetes.io/basic-auth \
     --from-literal=username=postgres --from-literal=password='${PASS}' \
     --dry-run=client -o yaml | kubectl apply -f -"
   #+END_SRC
2. Ensure the CR has =enableSuperuserAccess: true=. For prod, re-apply the
   updated manifest:
   #+BEGIN_SRC bash
   ssh beelink1 'cd ~/workspace/github/auth-manager && kubectl apply -f deploy/cnpg/pg-cluster.yaml'
   #+END_SRC
   For dev, re-running =scripts/deploy-beelink-dev.sh= applies the inline CR
   with the flag set (or =kubectl patch cluster platform-pg-dev -n
   auth-platform-dev --type=merge -p '{"spec":{"enableSuperuserAccess":true}}'=).
3. Let CNPG reconcile — it syncs the =postgres= role password to the secret.
   Verify =psql -U postgres -c 'SELECT 1'= works with the secret's password.

After this, drift stops and the LEGACY =ALTER USER= pre-steps + recovery runbook
are no longer needed for that cluster.

* Known gotchas / open issues

** #92 CNPG superuser drift on every redeploy (FIXED IN CODE — migration pending)
The =postgres= password used to be reset by CNPG on every reconcile because
=enableSuperuserAccess= defaulted to false (CNPG ≥1.21). Fixed in code via
=enableSuperuserAccess: true= + a pinned operator-owned basic-auth superuser
secret (=username=postgres=) — CNPG now syncs the role password to the secret
instead of rotating it. A ONE-TIME live-cluster migration is still required for
the existing prod + dev clusters (see "CNPG superuser pinning — one-time
migration" under Common ops). The LEGACY =ALTER USER= workaround remains only
until each cluster is migrated.

** #115 cf-setup-tunnel.sh wipes tenant ingress rules (CRITICAL)
Re-running the script destroys CloudflareTunnelService-added tenant rules
(=niralishappyfeetindia.com=, =safeandsoundhouses.com=, =maya-ai.ink=). Must
manually restore using the curl block in the Deployment section. Real fix:
read current ingress, MERGE static + tenant rules, preserve unknowns.

** #104 sendgrid_domain_valid doesn't persist after revalidate
Operators clicking "Re-verify" in the Email tab see "Authenticated ✓" but
the DB flag stays false. Subsequent save of From address rejected with
=email.unauthorized_from_domain=. Manual UPDATE on
=auth_manager.tenants.sendgrid_domain_valid= unblocks.

** #103 PermissionsTab error display
Operators get raw =ApiError: HTTP 400 {timestamp...}= JSON dumped on upload
failure. Should parse structured error code and show a friendly message.

** #113 deploy-beelink-dev.sh polish
First-install bugs: doesn't enable CNPG superuserAccess automatically, doesn't
seed =postgres-password= / =keycloak-admin-password= into the chart's secret.

** #114 theme-dev-sync.sh missing
Should script the kubectl cp loop for fast theme iteration in dev.

** #90 Wizard auto-verify admin email
After tenant create, operator must manually mark admin user's email verified.

** #94 IdP auto-link operator user
When operator-provisioned user later logs in via Google, KC creates a NEW
account instead of linking to the existing one. Manual fix per user today.
