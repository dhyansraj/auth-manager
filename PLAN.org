* Vision & Goals

** Problem statement

We need a *control plane* that lets a platform team onboard new customer
tenants, provision their apps (Keycloak clients), define their RBAC model, and
route incoming traffic to the correct tenant backend --- all without writing
ad-hoc scripts each time. Keycloak alone is not enough: it does not give us
declarative onboarding, an opinionated multi-tenant model, host-based routing,
audit, or a polished operator UI. =auth-manager= is the glue.

** Goals

1. *One-click tenant onboarding*: a single YAML manifest (or one UI form, or
   one =kubectl apply= against a Tenant CRD) provisions a new realm, a default
   set of clients, default groups, and registers the tenant's hostnames in the
   edge router.
2. *Idempotent declarative apps*: applying the same =AppManifest= twice is a
   no-op; mutating it produces a controlled diff and a version-tracked update.
3. *Source-of-truth split*: Keycloak owns identity; Postgres owns operational
   state (manifests, audit, jobs, routing intentions); Redis owns derived
   runtime state (host->tenant routing table, permission cache, JWKS cache).
4. *Verified JWTs everywhere*: every backend in the platform validates JWT
   signatures via JWKS (no exceptions; the legacy lib's signature-skip bug is
   fixed in =auth-lib v2=).
5. *Per-tenant isolation*: a security incident in tenant A must not be able to
   read tenant B's data, sessions, or tokens.
6. *Hybrid onboarding*: REST API for humans/UI, and a Kubernetes operator for
   GitOps-driven environments. Same backend code path.
7. *Local-first developer experience*: =make up= on a laptop yields a working
   end-to-end platform on k3d with trusted local TLS in under five minutes.

** Success criteria

- A new tenant with three apps and ten users can be created in < 60 seconds
  end-to-end via the UI.
- A login on tenant A's portal results in a JWT signed by tenant A's realm,
  verified by the backend using JWKS, with permissions cached in Redis on
  first call and served from cache for subsequent calls within token TTL.
- All onboarding actions appear in the audit log with =who=, =what=, =when=,
  =manifest_version=, =result=.
- Helm chart installs cleanly on k3d *and* on a managed Kubernetes cluster
  (EKS/GKE/AKS) without code changes --- only values overrides.

** Non-goals (explicit)

- *Not* a Keycloak replacement. We delegate all identity primitives to Keycloak.
- *Not* multi-region in v1. Single region, single Keycloak cluster, single
  Postgres primary. Geo-replication is a v2 concern.
- *Not* a billing/subscription system. Tenant metering belongs upstream.
- *Not* an identity provider broker UI in v1. Per-tenant SAML/OIDC federation
  can be configured in Keycloak directly; we expose it through the API later.
- *Not* a workflow engine. Onboarding jobs are linear and synchronous-ish;
  long-running approval workflows are out of scope.

** Inspiration: the JPM "PICO" pattern

The architectural shape (per-tenant realm + edge router + Redis routing table +
central onboarding control plane) is lifted from a prior platform that proved
itself at scale. The main delta in =auth-manager= is:

- *Cloudflare Tunnel* replaces the corporate ingress, routed by hostname
  (~Host~ header) rather than ~Referer~ --- ~Referer~ is unreliable,
  not always present, and trivially spoofable.
- *Helm-packaged* end to end so the same artifacts work on a laptop k3d and a
  production EKS cluster.
- *CRD + operator* alongside the REST API for GitOps workflows.

* Naming & Code Conventions

This section is the single source of truth for naming. Every new file in this
platform, and every renamed file during the migration, follows these rules.

** Java package root: =io.mcpmesh.auth.*=

All custom Java code uses the package root =io.mcpmesh.auth=. This is the
reverse of the =mcpmesh.io= domain, scoped to the =auth= subdomain to leave
room for sibling concerns later (e.g., =io.mcpmesh.mesh.*=, =io.mcpmesh.obs.*=).

~WHY:~ The existing repos use =ai.gentek.*= --- a holdover from an earlier
branding. Standardizing on =io.mcpmesh.*= now (before Phase 2) prevents the
mixed-namespace mess that comes from renaming half a codebase later. The cost
is a one-time IntelliJ "Refactor → Rename Package" per module, which is
mechanical and safe.

** Sub-package layout

| Module                  | Package root                                | Maven =groupId:artifactId=            |
|-------------------------+---------------------------------------------+---------------------------------------|
| Consumer library (v2)   | =io.mcpmesh.auth.lib.*=                     | =io.mcpmesh.auth:auth-lib=            |
| auth-manager backend    | =io.mcpmesh.auth.manager.*=                 | =io.mcpmesh.auth:auth-manager=        |
| tenant-operator         | =io.mcpmesh.auth.operator.*=                | =io.mcpmesh.auth:tenant-operator=     |

| Sample consumer app     | =io.mcpmesh.auth.client.sample.*=           | =io.mcpmesh.auth:auth-client-sample=  |

** Class naming

- *Drop org prefixes from class names.* The package already qualifies them.
  Auto-configuration classes are simply =SecurityAutoConfiguration= --- the
  fully qualified name =io.mcpmesh.auth.lib.security.SecurityAutoConfiguration=
  is unambiguous, and a =McpMesh= prefix would be noise.
- Spring =@Configuration= classes named =FooConfig= or =FooConfiguration=,
  consistent within a module (don't mix).
- Auto-configuration classes end in =AutoConfiguration= (Spring convention,
  picked up by detection in some tools).
- Test classes mirror the SUT name + suffix =Test= (unit) or =IT= (integration,
  Testcontainers-backed). Failsafe runs =*IT=, Surefire runs =*Test=.

re-rollout to add a tenant. With OpenResty + Redis, adding a tenant is a single
=SET host:bank1.your-org.com '{"tenant":"bank1","realm":"bank1","backend":"http://bank1-portal:8080"}'=
issued by =auth-manager= --- zero ingress churn, instant routing.

* Multi-Tenant Model & Keycloak Mapping

** Concept mapping

| Our concept                  | Keycloak primitive                                                | Notes                                                                                 |
|------------------------------+-------------------------------------------------------------------+---------------------------------------------------------------------------------------|
| Tenant                       | Realm                                                             | 1:1; deleting a tenant = deleting the realm + cascade in Postgres                     |
| App (a deployable service)   | Client (confidential, with =authorization-services-enabled=true=) | One client per app; ID = =<tenant>-<app>-client= for traceability                     |
| Role                         | Client role                                                       | Realm roles avoided --- keeps roles scoped to the app that owns them                  |
| Resource                     | Client resource (under authz settings)                            | Named like =order=, =invoice=                                                         |
| Scope                        | Client scope (under authz)                                        | Verb: =view=, =edit=, =delete=, =approve=                                             |
| Permission                   | Client resource-based permission                                  | Binds (resource, scope) -> role-policy. Emitted as authority =PERMISSION_ORDER_VIEW=  |
| User                         | Realm user                                                        | Created in the tenant's realm, never in =master=                                      |
| Group                        | Realm group                                                       | Used to bulk-assign client roles across apps within a tenant                          |
| Identity Provider (optional) | Realm IdP                                                         | Per-tenant SAML/OIDC federation; tenant brings their own corporate IdP                |
| Tenant admin                 | Realm role =realm-admin= in the tenant's realm                    | Distinct from platform admin which lives in =master=                                  |
| Platform admin               | Realm role in =master= realm                                      | Only this principal can call =/api/v1/tenants= (create/delete tenant)                 |

~WHY: realm-per-tenant rather than single-realm-with-groups.~ Three reasons
that override the "fewer realms = simpler ops" argument:

1. *Isolation.* A single misconfigured policy in a shared realm leaks across
   tenants. With realm-per-tenant, the blast radius of a misconfigured
   permission is one tenant.
2. *Independent identity providers.* Tenant A wants Okta, tenant B wants
   Azure AD, tenant C wants username/password. In a single realm you stack
   IdPs and rely on home-realm-discovery hacks. With realm-per-tenant, each
   realm trivially has its own IdP set.
3. *Clean deletion.* "Delete tenant" becomes "delete realm" + "delete schema
   rows tagged with tenant_id" --- both straightforward. In single-realm you
   must query-and-delete users by group membership, then re-evaluate policies,
   risking orphaned resources.

The cost: Keycloak does have a per-realm overhead. Empirically Keycloak 26
handles ~500 small realms on a 2-pod cluster comfortably; we will revisit if
we cross that.

* Component Designs

** Cloudflare Tunnel + DNS

External traffic enters the platform exclusively through Cloudflare Tunnel
(=cloudflared=). A wildcard DNS record =*.your-org.com= is proxied through
Cloudflare and routed via Tunnel to the in-cluster OpenResty service.

*** DNS pattern

- =admin.your-org.com= --- Admin UI (served by =auth-manager=)
- =auth-<tenant>.your-org.com= --- Keycloak realm endpoint for that tenant
  (proxied through OpenResty so we can rate-limit and audit)
- =<app>.<tenant>.your-org.com= --- the tenant's app
  (e.g., =portal.bank1.your-org.com=)

*** Wildcard TLS

In production: a Cloudflare-managed wildcard cert is fine; Tunnel terminates
TLS at Cloudflare and re-establishes TLS to the origin if configured.

Locally we use =mkcert= to mint a wildcard for =*.local.test= (see
[[*Local Kubernetes Setup (k3d)][Local Kubernetes Setup]]).

*** Tunnel ingress config

#+begin_src yaml
# ~/.cloudflared/config.yml on the bastion / or as a k8s ConfigMap
# mounted by the cloudflared deployment
tunnel: 8b1c8e1f-XXXX-XXXX-XXXX-XXXXXXXXXXXX
credentials-file: /etc/cloudflared/creds.json

ingress:
  # Send everything matching our domain to the cluster's openresty service
  - hostname: "*.your-org.com"
    service: https://openresty.auth-platform.svc.cluster.local:443
    originRequest:
      noTLSVerify: true   # internal cert; tighten in prod with originCAPool
      httpHostHeader: ""  # preserve original Host
      connectTimeout: 5s
  # Catch-all
  - service: http_status:404
#+end_src

*** Preventing tunnel bypass

~CAUTION:~ Anyone who learns the cluster's external IP could try to hit
OpenResty directly, bypassing Cloudflare's WAF and rate limits. Two defenses:

1. Restrict the cluster ingress LoadBalancer to Cloudflare's published IP
   ranges (https://www.cloudflare.com/ips/) via a cloud security group / k8s
   =LoadBalancerSourceRanges=.
2. In OpenResty, require =CF-Connecting-IP= header *and* validate the request
   originated from the Cloudflare IP set via =realip_module=.

** OpenResty edge router

OpenResty is nginx + LuaJIT + lua-resty-* libraries. We use it for one job:
look up the incoming =Host= in Redis, find the tenant + backend, and proxy.

*** Full annotated =nginx.conf=

#+begin_src nginx
# /etc/nginx/nginx.conf  (rendered from a ConfigMap; see Helm chapter)
worker_processes auto;
error_log /var/log/nginx/error.log notice;

events { worker_connections 4096; }

http {
    # ---- realip: trust Cloudflare so $remote_addr is the real client ------
    # Populated at startup by an init-container that curls
    # https://www.cloudflare.com/ips-v4 and renders this file.
    include /etc/nginx/cloudflare-realip.conf;
    real_ip_header CF-Connecting-IP;
    real_ip_recursive on;

    # ---- shared dict to cache Redis lookups (per-worker) -----------------
    lua_shared_dict tenant_cache 16m;
    lua_shared_dict locks 1m;

    # ---- Redis connection settings via env -------------------------------
    env REDIS_HOST;
    env REDIS_PORT;
    env REDIS_PASSWORD;

    # ---- structured access log in JSON for easy ingestion ----------------
    log_format json_combined escape=json
      '{"ts":"$time_iso8601","remote":"$remote_addr","host":"$host",'
      '"method":"$request_method","uri":"$request_uri","status":$status,'
      '"upstream":"$upstream_addr","rt":$request_time,"tenant":"$sent_http_x_tenant_id"}';
    access_log /var/log/nginx/access.log json_combined;

    # ---- rate limiting (per-IP, 50 r/s with 100 burst) -------------------
    limit_req_zone $binary_remote_addr zone=perip:10m rate=50r/s;

    # ---- the only server block --------------------------------------------
    server {
        listen 443 ssl http2;
        listen 80;
        server_name _;

        ssl_certificate     /etc/nginx/tls/tls.crt;
        ssl_certificate_key /etc/nginx/tls/tls.key;

        # Reject anything not from Cloudflare (production only)
        # set $cf_ok 0;
        # if ($http_cf_connecting_ip != "") { set $cf_ok 1; }
        # if ($cf_ok = 0) { return 403; }

        limit_req zone=perip burst=100 nodelay;

        # JWKS proxy: cache realm public keys at the edge so backends can
        # fetch them locally without each one hammering Keycloak.
        location ~ ^/realms/(?<realm>[^/]+)/protocol/openid-connect/certs$ {
            proxy_pass http://keycloak.auth-platform.svc.cluster.local:8080;
            proxy_cache jwks_cache;
            proxy_cache_valid 200 5m;
            proxy_cache_key "$host$uri";
        }

        # Main routing block
        location / {
            access_by_lua_file /etc/nginx/lua/route.lua;

    ngx.var.tenant_id      = rec.tenant_id
    ngx.var.tenant_realm   = rec.realm
    ngx.var.tenant_backend = rec.backend

    return ngx.exit(502)

    ngx.log(ngx.ERR, "malformed routing entry for ", host, ": ", raw)

ngx.var.tenant_realm   = rec.realm
ngx.var.tenant_backend = rec.backend

The =auth-manager= service publishes a Redis pub/sub event =routing.invalidate=
on tenant updates; OpenResty's =init_worker_by_lua= subscribes and proactively
flushes its per-worker cache. (Not shown above for brevity; included in
=routing-invalidator.lua=.)

~WHY: per-worker shared-dict cache.~ Each nginx worker keeps a 30-second
copy of the routing record. This caps Redis traffic at =workers * unique_hosts
/ 30s= regardless of QPS. Without it, every request would hit Redis.

** Keycloak

*** Version & deployment

- *Server*: =quay.io/keycloak/keycloak:26.6.1= (released 2026-04-15).
- *Admin Client (Java)*: =org.keycloak:keycloak-admin-client:26.0.5= --- note the
  admin client artifact lags the server release line significantly (server is on
  26.6.x while the admin client artifact published to Maven Central is on
  26.0.x). They are cross-compatible within the 26.x major.
- *Bitnami Helm chart =bitnami/keycloak=*: typically lags upstream by 1--3
  weeks. Pin =image.tag: "26.6.1"= explicitly in values.yaml rather than
  relying on the chart's bundled =appVersion=.
- *Database*: PostgreSQL 16 (KC 26.x supports 13+, we standardize on 16 for the
  whole platform --- see [[*Postgres][Postgres]]).
- *JVM*: Bitnami chart's bundled OpenJDK 21 is fine; do not override.
- *Replicas*: =2= for production, =1= for local. Postgres HA is the actual
  availability anchor --- KC nodes are stateless once cache is in DB.

*** Version pinning policy

Three rules. Violating any of these has cost us time on prior projects.

1. *Pin the patch version, not =:latest= or =:26=.* Keycloak has shipped
   ~weekly patches throughout the 26.x line in 2026, including two
   backward-incompatible admin-API changes (event listener SPI argument
   shapes, authz resource representation). Random rebuilds against =:latest=
   silently move you across these. Bump deliberately, with a changelog read.

2. *Server version and admin client version move independently.* The =org.keycloak:keycloak-admin-client= Maven artifact lags the server.
   Pin both explicitly in =pom.xml= and =values.yaml=. Don't assume bumping
   one implies the other.

3. *Drop =org.keycloak:keycloak-spring-boot-starter= --- it is deprecated.*
   Keycloak's official guidance since v25 is "use Spring Security's
   =spring-boot-starter-oauth2-resource-server= directly." The deprecated
   starter still resolves in Maven Central but receives no security fixes
   and pulls a stale Spring transitive tree. =auth-lib v2= drops it
   --- see [[*auth-lib v2 (formerly gentek-auth-lib)][auth-lib v2]]. If a copied =pom.xml=
   snippet brings it back, fail the build with a Maven enforcer rule.

*Bump cadence*: review for upgrade once per quarter. Read the release notes
for every minor between your current pin and the target before bumping; KC
release notes are short and high-signal.

~CAUTION:~ Never =:latest=. There is no LTS in community Keycloak --- each
minor is supported for ~6 months. If you need vendor LTS, use Red Hat Build
of Keycloak (RHBK) which ships ~6--9 months behind community.

*** Master realm bootstrap

The =master= realm holds:
- The platform admin user (=admin=).
- A *single* confidential client =auth-manager-admin-client= with
  service-account enabled and =realm-management= roles, used by =auth-manager=
  to provision per-tenant realms.

Credentials live in External Secrets, never in chart values.

*** Per-tenant realm template

When =auth-manager= creates a tenant, it imports a realm definition with these
defaults:

- =sslRequired= = =external=
- =loginTheme= = =auth-manager= (custom theme, optional)
- =eventsEnabled= = =true=, =eventsListeners= = [=jboss-logging=,
  =auth-manager-event-publisher=]
- =adminEventsEnabled= = =true=
- =bruteForceProtected= = =true=
- Token settings: =accessTokenLifespan= = 5m, =ssoSessionIdleTimeout= = 30m,

Each app onboarded under the tenant adds a client + resources + scopes +
permissions, reusing the existing =oidc-keycloak= code paths.

*** Custom event listener SPI

A small Keycloak SPI ships in =auth-manager/keycloak-spi/= that publishes
=LOGIN=, =LOGIN_ERROR=, =LOGOUT=, =USER_INFO_REQUEST=, and admin events to
=auth-manager='s =/internal/events= endpoint over an mTLS channel. The
backend writes those into =audit_events=.

** Postgres

*** One DB, two schemas

A single Postgres database, two logical schemas:
- =keycloak= --- owned by the =keycloak= role; used by the Keycloak pods.
- =auth_manager= --- owned by the =auth_manager= role; used by the backend.

~WHY: not separate databases.~ Backups, point-in-time-recovery, and connection
pooling get simpler with one DB; schema separation gives us the isolation we
actually want. We can split databases later if write contention forces it.

*** Connection pooling

PgBouncer in =transaction= pooling mode, deployed as a subchart. The
=auth-manager= backend uses HikariCP -> PgBouncer -> Postgres.

*** Migrations

Flyway, baseline at =V1__init.sql=. Migrations run as an init-container before
the =auth-manager= pod starts. Never run migrations from CI to prod; the
init-container handles it idempotently. See [[*Data Model][Data Model]] for the DDL.

** Redis

Single Redis (Bitnami subchart, with replica for HA). One Redis instance, four
logical key namespaces partitioned by prefix (since we are not yet at the
scale where separate clusters are warranted):

| Key pattern        | Value (JSON)                                              | TTL                  | Written by         | Read by              |
|--------------------+-----------------------------------------------------------+----------------------+--------------------+----------------------|
| =host:<hostname>=  | ={tenant_id, realm, backend}=                             | none (manual evict)  | =auth-manager=     | OpenResty            |
| =perm:<jti>=       | ={authorities:[...], exp}=                                | = token =exp= -now   | =auth-lib=         | =auth-lib=           |
| =jwks:<realm>=     | raw JWKS JSON                                             | 10 min               | =auth-lib=         | =auth-lib=           |
| =lock:<resource>=  | UUID token (=SET NX EX=)                                  | 30 sec               | =auth-manager=     | =auth-manager=       |
| =session:<sid>=    | RPT cache                                                 | session idle timeout | =auth-manager=     | =auth-manager=       |

*** Eviction & memory policy

=maxmemory-policy allkeys-lru=, with =maxmemory= sized at 2x expected hot
working set. The =host:*= keys carry =KEEPTTL= flag effectively by never being
written with EX; the LRU policy can evict them but they will be rehydrated on
the next request because OpenResty falls back to a 502 + a Postgres-driven
reconciliation alarm.

*** Service flow: =TenantService.create=

#+begin_src java
@Service
@RequiredArgsConstructor
public class TenantService {

    private final TenantRepository repo;
    private final RealmProvisioner realmProvisioner;
    private final RoutingTableWriter routing;
    private final AuditService audit;
    private final TransactionTemplate tx;

    public TenantResponse create(TenantCreateRequest req, String actor) {
        // 1. claim the tenant slug uniquely in Postgres (advisory lock + insert)
        TenantEntity entity = tx.execute(status -> {
            repo.advisoryLock(req.slug().hashCode());
            if (repo.existsBySlug(req.slug())) {
                throw new TenantAlreadyExistsException(req.slug());
            }
            return repo.save(TenantEntity.newPending(req));
        });

        try {
            // 2. provision the realm in Keycloak (idempotent w/ retry via resilience4j)
            String realmName = realmProvisioner.provision(req);

            // 3. write routing entries for each hostname
            req.hostnames().forEach(h ->
                routing.put(h, entity.getId(), realmName, req.defaultBackend()));

            // 4. mark active + audit
            tx.executeWithoutResult(s -> {
                entity.markActive(realmName);
                audit.record(actor, "tenant.create", entity.getId(), req);
            });
            return TenantResponse.from(entity);

        } catch (Exception e) {
            tx.executeWithoutResult(s -> {
                entity.markFailed(e.getMessage());
                audit.record(actor, "tenant.create.failed", entity.getId(), req);
            });
            throw e;
        }
    }
}
#+end_src

~WHY: claim-then-provision pattern.~ We commit a =PENDING= row first so two
concurrent =POST /tenants= for the same slug cannot both call Keycloak.
If Keycloak provisioning fails, the row stays as =FAILED= and is visible in
audit; a retry or manual cleanup is a single API call.

** auth-lib v2 (formerly gentek-auth-lib)

This section specifies =auth-lib v2= (the port of the legacy =gentek-auth-lib=).
This is the *most security-critical* change in the entire project.

*** What's wrong in v1

#+begin_quote
~BLOCKER:~ v1 ships a =JwtDecoder= that returns a =Jwt= built from the raw
unverified token. *Any signature is accepted; even tokens with =alg: none= are
accepted.* This was almost certainly added during local development to bypass
JWKS connectivity issues and never removed.
#+end_quote

The v2 decoder must:

1. Fetch JWKS from =$\{keycloak.url\}/realms/$\{keycloak.realm\}/protocol/openid-connect/certs=.
2. Verify signature with the matching =kid=.
3. Refresh JWKS on key rotation (cache-control or 5-minute TTL, whichever is shorter).
4. Reject =alg: none= explicitly.
5. Validate =iss=, =exp=, =nbf=, and (for clients that pass it) =aud=.

#+begin_src java
package io.mcpmesh.auth.lib.security;

import org.springframework.data.redis.core.StringRedisTemplate;
import org.springframework.security.core.GrantedAuthority;
import org.springframework.security.core.authority.SimpleGrantedAuthority;
import org.springframework.security.oauth2.jwt.Jwt;
import org.springframework.web.client.RestTemplate;

import java.time.Duration;
import java.time.Instant;
import java.util.*;

public class PermissionService {

    private final StringRedisTemplate redis;
    private final RestTemplate http;
    private final AuthProperties props;
    private final ObjectMapper json = new ObjectMapper();

    public PermissionService(StringRedisTemplate redis,
                             RestTemplate http,
                             AuthProperties props) {
        this.redis = redis;
        this.http = http;
        this.props = props;
    }

    public Collection<GrantedAuthority> authoritiesFor(Jwt jwt) {
        String jti = jwt.getId();
        String cached = redis.opsForValue().get("perm:" + jti);
        if (cached != null) {
            return decode(cached);
        }

        // UMA ticket exchange: ask Keycloak for the RPT for this token
        String tokenEndpoint = props.url() + "/realms/" + props.realm()
                             + "/protocol/openid-connect/token";

        var headers = new HttpHeaders();
        headers.setContentType(MediaType.APPLICATION_FORM_URLENCODED);
        headers.setBearerAuth(jwt.getTokenValue());

        var form = new LinkedMultiValueMap<String, String>();
        form.add("grant_type", "urn:ietf:params:oauth:grant-type:uma-ticket");
        form.add("audience", props.clientId());
        form.add("response_mode", "permissions");

        var resp = http.exchange(tokenEndpoint, HttpMethod.POST,
                                 new HttpEntity<>(form, headers),
                                 new ParameterizedTypeReference<List<UmaPermission>>() {});
        List<GrantedAuthority> auths = resp.getBody().stream()
            .flatMap(p -> p.scopes().stream()
                          .map(s -> "PERMISSION_" + p.rsname().toUpperCase()
                                  + "_" + s.toUpperCase()))
            .distinct()
            .map(SimpleGrantedAuthority::new)
            .collect(Collectors.toUnmodifiableList());

        long ttl = Math.max(1, jwt.getExpiresAt().getEpochSecond() - Instant.now().getEpochSecond());
        redis.opsForValue().set("perm:" + jti, encode(auths), Duration.ofSeconds(ttl));
        return auths;
    }
    // encode/decode helpers omitted for brevity
}

*** Reconciler skeleton

#+begin_src java
@ControllerConfiguration
public class TenantReconciler implements Reconciler<Tenant>, ErrorStatusHandler<Tenant> {

    private final AuthManagerClient api; // calls /api/v1/tenants*

    public TenantReconciler(AuthManagerClient api) { this.api = api; }

    @Override
    public UpdateControl<Tenant> reconcile(Tenant resource, Context<Tenant> ctx) {
        var spec = resource.getSpec();
        String hash = ManifestHasher.sha256(spec);
        var status = resource.getStatus() == null ? new TenantStatus() : resource.getStatus();

        if (hash.equals(status.getLastReconciledHash()) && "Ready".equals(status.getPhase())) {
            return UpdateControl.noUpdate(); // converged
        }

        status.setPhase("Provisioning");
        UpdateControl<Tenant> intermediate = UpdateControl.patchStatus(resource);

        var apiResp = api.applyTenant(spec);  // POST /api/v1/tenants:apply
        status.setRealm(apiResp.realm());
        status.setLastReconciledHash(hash);
        status.setPhase("Ready");
        return UpdateControl.patchStatus(resource);
    }

    @Override
    public ErrorStatusUpdateControl<Tenant> updateErrorStatus(Tenant r, Context<Tenant> c, Exception e) {
        var s = r.getStatus() == null ? new TenantStatus() : r.getStatus();
        s.setPhase("Failed");
        s.addCondition("Reconciled", "False", "ApplyFailed", e.getMessage());
        return ErrorStatusUpdateControl.patchStatus(r);
    }
}
#+end_src

* End-to-end Flows

** Tenant onboarding (UI / YAML / CRD)

Sequence (all three input paths converge at step 2):

1. *Input*:
   - UI: admin fills the "Create Tenant" form -> =POST /api/v1/tenants=.
   - CLI: =curl -d @tenant.yaml ... /api/v1/tenants:apply=.
   - GitOps: =kubectl apply -f tenant.yaml= (CRD) -> operator -> same REST API.
2. =auth-manager= authenticates the caller (platform admin role required).
3. =TenantService.create= claims slug with an advisory lock + inserts a
   =PENDING= row in =tenants=.
4. =RealmProvisioner= calls Keycloak admin REST: create realm, apply realm
   template, create default groups, install event listeners.
5. =RoutingTableWriter= writes =host:<hostname>= entries in Redis for each
   tenant hostname.
6. Postgres row flipped to =ACTIVE=; audit event recorded.
7. Pub/sub: =auth-manager= publishes =routing.invalidate= so OpenResty drops
   its per-worker cache.
8. Response returned (UI shows success; operator updates =.status.phase=Ready=).

** App onboarding within a tenant

1. Admin posts =AppManifest= YAML to =POST /api/v1/tenants/{id}/apps=.
2. =AppOnboardingService= computes a content hash, checks for an existing
   =app_manifests= row with the same hash + tenant + app slug --- if equal,
   returns =200 OK= with =changed=false= (true idempotency).
3. Otherwise: calls the (ported) =oidc-keycloak= code path:
   create client, roles, resources, scopes, role-based authz policies.
4. Inserts a new =app_manifests= row with version = max(version)+1 and the
   new hash + raw YAML.
5. Audit event recorded.

** User onboarding & group assignment

1. =POST /api/v1/tenants/{id}/users= with =\{username, email, temporaryPassword\}=.
2. =UserProvisioner= creates the user in the tenant's realm, sets a temporary
   password with =requiredActions=[UPDATE_PASSWORD]=.
3. Optional: =POST /api/v1/tenants/{id}/users/{uid}/groups= adds the user to
   groups, which through Keycloak's group-role mappings grants client roles.
4. =user_cache= row in Postgres updated for fast filtering in the audit UI.

** Runtime auth: bank1 portal login

1. Browser hits =https://portal.bank1.your-org.com/=.
2. Cloudflare Tunnel forwards to OpenResty, preserving =Host: portal.bank1.your-org.com=.
3. OpenResty Lua: =GET host:portal.bank1.your-org.com= from Redis -> finds
   ={tenant_id: ..., realm: "bank1", backend: "http://bank1-portal:8080"}=.
4. OpenResty injects =X-Tenant-Id= + =X-Realm= and =proxy_pass='es.
5. The portal Spring Boot app (built with =auth-lib v2=) sees no JWT,
   redirects the browser to
   =https://auth-bank1.your-org.com/realms/bank1/protocol/openid-connect/auth?...=.
6. Cloudflare/OpenResty routes that to Keycloak. User logs in. Keycloak issues
   an authorization code; the portal exchanges it for an access token.
7. Subsequent request carries =Authorization: Bearer <jwt>=.
8. =NimbusJwtDecoder=:
   - Fetches JWKS for =bank1= (cached for 5 minutes, served from OpenResty's
     JWKS proxy cache for 5 minutes on top of that).
   - Verifies signature with matching =kid=.
   - Validates =iss= = =https://auth-bank1.your-org.com/realms/bank1=.
9. =KeycloakPermissionService= looks up =perm:<jti>= in Redis. Miss -> UMA
   request to Keycloak; hit -> sub-ms response.
10. Authorities (e.g., =PERMISSION_ACCOUNT_VIEW=) attached to the Authentication.
11. =@Secured("PERMISSION_ACCOUNT_VIEW")= on the controller enforces access.

** Tenant suspension & deletion

Suspension (reversible):
- =PATCH /api/v1/tenants/{id}= with =\{"suspended": true\}=.
- Keycloak realm =enabled=false= (immediately rejects logins and token refresh).
- Redis =host:<h>= entries' =backend= updated to a "service unavailable" stub
  (or removed; with removal, OpenResty returns 404).
- Postgres status -> =SUSPENDED=.
- Audit event recorded.

Deletion (destructive, requires =?cascade=true=):
- Realm deleted in Keycloak.
- All =host:*= entries for the tenant cleared from Redis; =perm:*= for tokens
  issued by that realm naturally expire.
- Postgres rows soft-deleted (tombstone) *except* =audit_events= which retains
  forever for compliance.
- Audit event recorded.

* Security Model

** JWT signature verification

~BLOCKER REMEDIATION:~ =auth-lib v2= uses
=NimbusJwtDecoder.withJwkSetUri(...)= which:
- Fetches the realm's JWKS via HTTPS.
- Caches it locally (5 minutes by default; honors =Cache-Control= if shorter).
- Selects the verification key by =kid= header.
- Rejects =alg: none= unconditionally.
- Rejects unknown =kid= (forcing a JWKS refresh first; if still unknown -> 401).

JWKS rotation cadence: Keycloak rotates RSA signing keys quarterly by default.
A 5-minute JWKS cache means worst case 5 minutes of 401s after rotation if a
backend was holding the old key; in practice the NimbusJwtDecoder triggers a
refresh on unknown =kid=, so the user-visible failure window is one request.

** Permission cache invalidation

Cache key: =perm:<jti>=. TTL: =min(configured cap, token.exp - now)=. We never
cache past the token's own expiration.

For role changes: a Keycloak admin who changes a user's roles forces re-login
in v1 (token still carries old permissions until expiry). In v2 we can add a
Keycloak event listener that publishes =USER_GROUP_MEMBERSHIP= /
=ROLE_ASSIGNMENT= events and invalidates =perm:*= for active tokens owned by
that user. See Open Questions for the trade-off.

** Per-tenant isolation

- Realm-per-tenant (see [[*Multi-Tenant Model & Keycloak Mapping][Multi-Tenant Model & Keycloak Mapping]]).
- =tenant_id= is a column on every operational table; all queries from the

- Tenant backends are deployed in their own k8s namespaces
  (=tenant-<slug>=). =NetworkPolicy=:
  - ingress only from =auth-platform= namespace (OpenResty) and same-namespace.
  - egress only to =auth-platform/keycloak= and =auth-platform/redis=.
- Redis cache keys are tagged with realm/tenant where ambiguous (=perm:<jti>=
  is implicitly tenant-scoped because the JTI is unique across realms).

** Audit logging

Two sources:
1. =auth-manager= writes every state-changing API call to =audit_events=
   (action, actor, target, request payload hash, status, timestamp).
2. Keycloak's custom event listener SPI posts to =/internal/events= for
   identity events (LOGIN, LOGIN_ERROR, LOGOUT, REGISTER, UPDATE_PASSWORD,
   admin events).

Retention: 1 year hot in Postgres, archival to object storage after that
(s3-csi-driver-style export job; not in v1).

** CSRF / CORS for admin UI

Because the SPA is served from the same origin as the API, CORS is not needed
in production. For local dev where Vite runs on 5173 and the API on 9090, we
enable CORS for =http://localhost:5173= only and behind an
=app.env=local= property.

CSRF: APIs are stateless and authenticated by bearer token, so Spring's
default CSRF protection is disabled. For *future* form-based admin endpoints
we would re-enable it with a synchronizer token.

** Rate limiting at OpenResty

- Per-IP: 50 r/s, burst 100 (already in =nginx.conf= example above).
- Per-tenant token endpoint: 10 r/s/IP on =/realms/*/protocol/openid-connect/token=
  to slow brute-force.
- Admin API: 20 r/s/IP on =admin.your-org.com=.

For prod we add a Cloudflare WAF rule above this for L7 DDoS.

* IdP Brokering Strategy

** Terminology: brokering vs federation

Keycloak's documentation uses two terms that are easy to conflate:

- *Identity Brokering* --- Keycloak delegates the authentication step to an
  external OIDC or SAML IdP (Google, Microsoft Entra ID, GitHub, another
  Keycloak). The external IdP returns an assertion; Keycloak validates it,
  provisions or looks up a local user, and mints its *own* token. This is
  what we want.
- *User Federation* --- Keycloak reads the user store itself from an
  external system (LDAP, Kerberos, custom SPI). Passwords may be verified
  against the remote store; the user record itself lives elsewhere. This is
  a different mechanism with different operational characteristics.

~NOTE:~ The industry routinely uses "federation" to mean either. This
document uses *brokering* throughout when we mean the OIDC-delegation
pattern; *federation* is reserved for the user-store integration pattern,
which we do not use in v1.

** Current state (the seven Google OAuth clients problem)

The brownfield reality today: every app authenticates users by being its
own OIDC client of Google. Each app owns a distinct Google Cloud OAuth
client ID/secret and its own redirect URI registration.

| App        | Auth front                    | Google OAuth client |
|------------+-------------------------------+---------------------|
| ai-debate  | dedicated nginx oauth2_proxy  | client #1           |
| aicareers  | dedicated nginx oauth2_proxy  | client #2           |
| dropwatch  | dedicated nginx oauth2_proxy  | client #3           |
| happyfeet  | =auth-gateway= (custom)       | client #4           |
| maya       | dedicated nginx oauth2_proxy  | client #5           |
| skybot     | dedicated nginx oauth2_proxy  | client #6           |
| wisefolio  | dedicated nginx oauth2_proxy  | client #7           |

Problems this creates:
- Seven OAuth clients to rotate when a secret leaks.
- Seven sets of redirect URIs to maintain in Google Cloud Console.
- No SSO across apps --- logging into ai-debate does nothing for skybot.
- Per-app session cookies; logging out of one app does not log out of any
  other.
- No centralized view of who the users are, what they have permission to
  do, or when they last signed in.
- No way to swap Google for another IdP (Microsoft, GitHub, a corporate
  SAML) without touching every app's code and config.

** Target state

#+begin_example
   User                  app1                Keycloak              Google

What this buys us:
- *ONE* Google OAuth client (the one Keycloak owns). Six fewer secrets to
  rotate, six fewer redirect URI registrations.
- True SSO across all apps in the same realm.
- IdPs become per-realm configuration data. Adding GitHub login for one
  tenant is a manifest change, not a code change in any app.
- Per-tenant domain restriction (Google Workspace =hostedDomain=, Entra ID
  tenant pinning).
- Local Keycloak users and brokered users coexist in the same realm ---
  useful for break-glass admins and contractors without corp accounts.
- Permissions evaluated once, in Keycloak, regardless of how the user
  authenticated.

~WHY:~ We chose brokering over keeping per-app OIDC because (a) it
collapses the Google Cloud Console sprawl to a single client, (b) it
delivers SSO with no app-side work, (c) it decouples app code from any
specific IdP --- the apps only ever know Keycloak --- and (d) per-realm
IdP configuration lets tenant A authenticate against Google Workspace
=bank1.com= while tenant B uses Microsoft Entra ID, with neither app
caring. Alternative considered: keep per-app Google OAuth and just
standardize the client configuration via a shared Terraform module ---
rejected because it doesn't deliver SSO and still leaves seven secrets
to rotate.

** Per-tenant IdP configuration

Each Keycloak realm is one tenant and owns its own IdP configuration set.
Realm =bank1= can broker to Google Workspace =bank1.com=; realm =bank2=
can broker to Microsoft Entra ID; realm =coop1= can broker to both plus
local password login. Apps never see any of this --- they redirect to
their realm's =/auth= endpoint and consume whatever Keycloak gives back.

The =auth-manager= tenant manifest extends with an =identityProviders=
block, which the operator turns into Keycloak Admin API calls:

#+begin_src yaml
tenant:
  id: bank1
  hostname: bank1.example.com
realm:
  displayName: "Bank 1"
identityProviders:
  - alias: google
    providerId: google
    enabled: true
    config:
      clientId: ${BANK1_GOOGLE_CLIENT_ID}
      clientSecret: ${BANK1_GOOGLE_CLIENT_SECRET}
      hostedDomain: bank1.com      # CRITICAL: restricts to this Google Workspace
      defaultScope: "openid email profile"
      syncMode: FORCE              # always update user profile from Google
  - alias: microsoft
    providerId: oidc
    enabled: true
    config:
      authorizationUrl: https://login.microsoftonline.com/<tenant-id>/oauth2/v2.0/authorize
      tokenUrl: https://login.microsoftonline.com/<tenant-id>/oauth2/v2.0/token
      clientId: ${BANK1_MS_CLIENT_ID}
      clientSecret: ${BANK1_MS_CLIENT_SECRET}

~CAUTION:~ Always set =hostedDomain= for Google IdPs in B2B realms.
Without it, *any* Google account can attempt the login --- Keycloak will
ultimately reject on user-not-found (or auto-create, depending on policy),
but you have already advertised that your IdP exists and accepts arbitrary
Google identities. That is a phishing signal you do not want to emit.

** First Login Flow --- the email-verification trap

Keycloak's First Login Flow decides what happens the first time a brokered
identity appears in a realm. The three practical policies:

1. *Auto-create* (default): create a fresh local user from the IdP claims.
   Safe so long as the IdP attests =email_verified=true=.
2. *Auto-link by email*: if a local user with the same email already
   exists, transparently link the new IdP identity to that user.
   Convenient and *dangerous when =email_verified=false=*: an attacker
   registers a Google consumer account claiming the victim's email
   (unverified by Google), logs in via Keycloak, and gets attached to the
   victim's real account. Account takeover.
3. *Review profile* / *Require admin approval*: human-in-the-loop
   confirmation. Slow but auditable; appropriate for high-assurance
   environments.

~BLOCKER:~ Whichever policy you pick, configure the IdP definition to
*require* =email_verified=true= from the upstream token. On the IdP's
"Advanced" tab in the Keycloak admin console, set *Trust Email* only if
the upstream IdP guarantees verification:
- Google Workspace: yes.
- Google consumer accounts: mixed --- the field is populated but not
  always meaningful for accounts created with arbitrary email addresses.
- Microsoft Entra ID (work accounts): yes.
- Microsoft consumer accounts: no.

When in doubt, leave *Trust Email* off and let Keycloak's own email
verification flow run.

~WHY:~ The default this project picks is *auto-create + require
=email_verified=true= + do not auto-link by email*. If a user already has
a local account and later wants to use Google, they must explicitly use
the "Link account" action from their Keycloak account console ---
preventing the silent-takeover vector entirely.

** Token format change (the actual migration work)

Before migration, apps validate Google ID tokens:
- =iss = https://accounts.google.com=
- =aud = <that-app's-Google-OAuth-client-id>=
- JWKS from =https://www.googleapis.com/oauth2/v3/certs=

After migration, apps validate Keycloak-issued tokens:
- =iss = https://kc.dhyanraj.dev/realms/<tenant-id>=
- =aud = <keycloak-client-id-for-this-app>=
- JWKS from =https://kc.dhyanraj.dev/realms/<tenant-id>/protocol/openid-connect/certs=

For Spring Boot apps using =auth-lib v2= this is a configuration
change, not a code change:

#+begin_src properties
keycloak.url=https://kc.dhyanraj.dev
keycloak.realm=tenant1
keycloak.client-id=app1
#+end_src

The =NimbusJwtDecoder.withJwkSetUri(...)= machinery (see
[[*JWT signature verification][JWT signature verification]]) does the rest --- JWKS fetch, kid-based key
selection, cache, refresh-on-unknown-kid.

For nginx apps still fronted by =oauth2_proxy=, the equivalent config
swap:

#+begin_src conf
provider = "oidc"
oidc_issuer_url = "https://kc.dhyanraj.dev/realms/tenant1"
client_id = "app1"
client_secret = "${APP1_CLIENT_SECRET}"
email_domains = ["*"]   # let Keycloak enforce, not oauth2_proxy
#+end_src

~NOTE:~ =oauth2_proxy= at the edge becomes optional once OpenResty +
tenant-scoped routing is in place --- see [[*OpenResty edge router][OpenResty edge router]] for the
recommended "auth check via OpenResty Lua + introspection" alternative
that lets us delete =oauth2_proxy= from the per-app pods entirely.

** Logout, SSO, and session management

With brokering, Keycloak owns the user session; the apps do not. The
behavioural consequences:

- *Single Sign-On*: user logs into app1 --- the browser receives a
  Keycloak session cookie scoped to =kc.dhyanraj.dev=. When the same
  browser visits app2 and gets redirected to Keycloak, Keycloak sees the
  existing session and immediately redirects back with a fresh token. No
  second password prompt, no second Google round-trip.
- *Single Logout (SLO)*: hitting
  =/realms/<tenant>/protocol/openid-connect/logout= ends the Keycloak
  session and triggers back-channel logout notifications to every app the
  user is currently logged into (each app having registered a
  =backchannel_logout_uri= as part of its client config).
- *Token lifetime tuning*: short access-token TTL (5--15 min), longer
  refresh-token TTL (hours to days), refresh-token rotation on. Apps
  either use refresh tokens themselves or simply re-redirect to Keycloak
  on access-token expiry --- the Keycloak session cookie makes the
  redirect silent.
- *IdP-initiated logout*: if Google revokes a user's session out-of-band,
  Keycloak does *not* learn this automatically. The upper bound on a
  stale-session window is the access-token TTL. If your threat model
  requires faster propagation, enable Google Admin SDK push notifications
  and consume them in a Keycloak Event Listener SPI; otherwise rely on
  the TTL.

** Trust and failure modes

When *Google* is down:
- New logins via Google fail. Existing Keycloak access tokens remain valid
  until their =exp=.
- Mitigation: keep at least one local Keycloak admin user per realm (no
  IdP attached) for break-glass.

When *Keycloak* is down:
- New logins fail entirely. JWKS fetches fail --- but apps with cached
  JWKS keep validating existing tokens until cache expiry (recommended
  cache: 10 min). Apps that re-fetch JWKS on every request (an
  anti-pattern, do not do this) fail immediately.
- Mitigation: run Keycloak with at least two replicas behind Postgres HA.
  =auth-lib v2='s JWKS cache absorbs short outages.

When a *tenant's IdP is misconfigured* (e.g., wrong client secret rotated
into Google):
- Only that realm's logins fail; other realms are unaffected. This is the
  strong-isolation argument for realm-per-tenant --- see
  [[*Multi-Tenant Model & Keycloak Mapping][Multi-Tenant Model & Keycloak Mapping]].

** Migration ordering (per-app)

For each existing app, the per-app cutover sequence is:

1. Create an OIDC client in the target tenant's realm. Set *Valid Redirect
   URIs* to the app's hostname(s).
2. Add the Google IdP to the realm (one-time per realm; idempotent on
   re-runs of the operator).
3. In the app, either upgrade to =auth-lib v2= (Java apps) or
   repoint =oauth2_proxy= (nginx apps) --- see
   [[*Token format change (the actual migration work)][Token format change (the actual migration work)]].
4. Deploy the migrated app side-by-side with the current Google-OAuth
   version on a parallel hostname (e.g., =skybot-kc.dhyanraj.dev=).
5. Validate: login works, permission claims are correct, logout works,
   token refresh works.
6. Cut DNS / Cloudflare Tunnel ingress over from the old hostname to the
   new.
7. Delete the app's old OAuth client in Google Cloud Console.
8. Repeat for the next app.

Per-app cutover effort: *roughly half a day* for a typical Spring Boot
app (config change + redeploy + smoke test), *roughly one day* for nginx
+ custom-auth apps like =happyfeet='s =auth-gateway= (which is being
retired wholesale rather than reconfigured).

** Tech notes

- *TanStack Query* for all server state. Mutations call =queryClient.invalidateQueries=.
- *Zustand* for UI-only state: sidebar collapsed, command palette open, etc.
- *react-hook-form + zod* for forms; zod schemas are co-located with API DTOs.
- *Monaco* loaded lazily (=React.lazy=) because it's ~2 MB.
- *Auth* via =oidc-client-ts=; redirect to the =master= realm for login, store
  the access token in memory only (refresh token in httpOnly cookie issued by
  the backend after the auth-code callback).

** Umbrella vs library vs application charts

- *Application chart* (=auth-platform=): the deployable unit. Templates render
  actual k8s resources.
- *Library chart*: pure helpers (= named templates, no rendered resources).
  We extract one (=auth-platform-lib=) once we duplicate enough boilerplate.
- *Umbrella*: an application chart whose primary job is composing subcharts.
  =auth-platform= is an umbrella in spirit but ships its own templates too;
  this is fine.

Use semver on the chart (=0.1.0= -> =0.2.0= for backwards-incompat values
changes). =appVersion= tracks the app image version but is informational.

Hooks: use =helm.sh/hook: pre-install,pre-upgrade= for the migration init only
if we move it out of the Deployment initContainer. Generally we prefer the
initContainer approach because it ties migration to pod startup, simpler to
reason about during rollbacks.

* Local-First Development Loop

The fastest feedback loop for daily development is *not* Kubernetes. Helm
upgrades, image builds, and pod restarts add 30--60 seconds per code change ---
unacceptable for iterating on a Spring controller or a React component. The
local-first loop runs dependencies in Docker Compose and the apps themselves
directly from IDE/CLI with hot reload.

** Three layers of "local"

| Layer | Purpose                                         | Tooling                       | Code change → feedback |
|-------+-------------------------------------------------+-------------------------------+------------------------|
| 1     | Daily dev: backend, UI, lib, controller logic   | Docker Compose + =mvn= + =npm=| <1 second              |
| 2     | Integration: operator, CRDs, Helm, networking   | k3d                           | 30--60 seconds         |
| 3     | Pre-prod soak: real DNS, CF Tunnel, real TLS    | beelink k3s cluster           | minutes                |

~WHY:~ Each layer is strictly more realistic but strictly slower. Spend ~80% of
dev time in Layer 1, ~15% in Layer 2 (when touching the operator or Helm),
~5% in Layer 3 (before opening a hostname to the world). Most projects skip
Layer 1 and hate their dev loop; we won't.

Skip Layer 1 ONLY for: =tenant-operator= (needs a real cluster), Helm chart
template debugging (needs a cluster), NetworkPolicy/RBAC behavior (needs a
cluster). Everything else belongs in Layer 1.

** When to graduate to Layer 2 (k3d)

- About to write the =tenant-operator= reconciler (CRDs need a real cluster).
- Helm chart change you want to render and apply end-to-end.
- Multi-pod behavior --- KC clustering, Redis Sentinel, Postgres replica
  failover.
- NetworkPolicy or RBAC change.
- Final smoke test before opening a PR to =main=.

Don't graduate for: routine backend code changes, UI work, lib changes, config
tweaks. Those stay in Layer 1.

** Common pitfalls

- *Forgot to set =SPRING_PROFILES_ACTIVE=local=*. App tries to connect to
  Bitnami-style URLs (=keycloak.auth-platform.svc.cluster.local=) and dies.
  Fix: set the profile in the IDE run config once.
- *Vite proxy 502 to backend*. Spring app isn't running yet, or =mvn= is still
  compiling. Wait, refresh.
- *KC realm.json import fails*. The realm already exists. Set
  =--override=true= in =make dev-seed=, or =docker compose down -v= then
  =make dev=.
- *Postgres "database authmanager does not exist"*. The
  =postgres-init.sh= script didn't run because you mounted compose with stale
  volumes from before the init script existed. Solution: =make dev-clean= then
  =make dev=.
- *OpenResty can't reach Redis*. Service name typo in =nginx.conf= or
  =route.lua= --- use =redis= (compose service name), not =localhost=, since
  OpenResty is inside the compose network.
- *Tests pass locally but fail in CI*. Likely a Layer-1 vs Layer-2 difference.
  CI runs Testcontainers --- a deeper match to production than compose. If
  this happens, run the test suite against k3d before pushing.

** Reproducing the local stack in CI?

Don't. CI uses Testcontainers (Layer-2-equivalent in-process) which is more
representative for tests than a hand-rolled compose. The compose stack exists
for *human* feedback loops; the test stack exists for *machine* feedback loops.
Different tools, both correct for their jobs.

** =oidc-keycloak= -> =auth-manager=

| =oidc-keycloak= file                          | Fate                                                                             |
|-----------------------------------------------+----------------------------------------------------------------------------------|
| =AppRequest.java= (DTO)                       | Lifted as-is into =auth-manager.domain.AppManifest=                              |

** legacy lib v1 -> =auth-lib= v2

Breaking changes:

1. =@ComponentScan("ai.gentek.oidc.common")= no longer required (auto-config
   picks up the new =io.mcpmesh.auth.lib.*= classes).
2. Properties moved from arbitrary @Value injection to =@ConfigurationProperties(prefix="keycloak")=.
3. JWT signature is now *verified* --- any consumer relying on accepting
   forged tokens (none should, but...) will see 401s.
4. Redis dependency added; consumers must provide a Redis or set
   =keycloak.permission-cache.enabled=false=.
5. *Package rename*: all symbols move to =io.mcpmesh.auth.lib.*=. See
   [[*Naming & Code Conventions][Naming & Code Conventions]] for the full migration mapping.

Migration steps for a consumer app (the sample consumer formerly known as
=auth-client=, retargeted as =auth-client-sample=):
1. Bump dependency to =2.0.0= (=groupId= becomes =io.mcpmesh.auth=,
   =artifactId= becomes =auth-lib=).
2. Remove =@ComponentScan("ai.gentek.oidc.common")= from the main class.
3. Update import statements to =io.mcpmesh.auth.lib.*=. IntelliJ "Optimize
   Imports" does this automatically after the lib dependency is bumped.
4. Add to =application.yml=:
   #+begin_src yaml
   keycloak:
     url: ${KEYCLOAK_URL}
     realm: ${KEYCLOAK_REALM}
     client-id: ${KEYCLOAK_CLIENT_ID}
     client-secret: ${KEYCLOAK_CLIENT_SECRET}
   spring:
     data:
       redis:
         host: ${REDIS_HOST}
   #+end_src
5. Set =KEYCLOAK_URL= to the realm issuer URL exactly as Keycloak returns in
   =iss= (e.g., =https://auth-bank1.your-org.com= --- NOT the internal service
   URL, because issuer comparison is exact-match).
6. Re-deploy. If 401s appear immediately, check JWKS reachability from the pod.

* Open Questions / Decisions to Revisit

- *Postgres-per-tenant vs shared with schema separation?*
  Currently shared. Revisit if: any tenant requires PII residency in a
  specific region, OR a single tenant's write rate dominates the others, OR
  we cross ~50 tenants and per-tenant backup/restore becomes operationally
  attractive.

- *Keycloak vs Authentik vs ZITADEL?*
  Sticking with Keycloak. Rationale: most mature admin REST API, strongest
  community, our existing code targets it, UMA support is best-in-class.
  Authentik has a nicer UI but a weaker admin API. ZITADEL is API-first but
  the multi-tenancy model is different enough that switching would mean
  rewriting realm-provisioning. Revisit only if Keycloak's Quarkus migration
  causes us pain we can't mitigate.

- *Operator language: =java-operator-sdk= vs =kubebuilder=?*
  Default Java for stack uniformity (one toolchain, one IDE, shared DTOs).
  Cost: ~250 MB image vs ~30 MB. Revisit if the operator becomes a
  bottleneck or if we add several more controllers (a Go operator project
  starts to amortize better at that scale).

- *Push-based permission invalidation vs TTL-only?*
  Start with TTL (=perm:<jti>= TTL = token =exp=). Add Keycloak event
  listener push-invalidation in Phase 5 if/when role-change-latency
  complaints emerge. Note: push invalidation also requires being able to
  enumerate active =jti= values per user, which we get by also writing a
  =user:<sub>:tokens= set in Redis on permission cache fill.

- *Single Postgres vs CloudNativePG operator?*
  Single Bitnami chart for v1. CNPG gives us HA + PITR with less work but
  adds an operator to the dependency graph. Move to CNPG in Phase 5.

- *Front-end auth: oidc-client-ts vs server-side BFF pattern?*
  Currently planning oidc-client-ts (SPA-driven PKCE). The BFF (Backend for
  Frontend) pattern --- where the SPA gets an httpOnly session cookie and the
  backend talks to Keycloak --- is more secure (no tokens in the browser).
  Consider switching in Phase 3 review.

- *Should the Keycloak SPI be in this repo or its own?*
  In this repo (=keycloak-spi/= submodule) for v1 to avoid version skew.
  Revisit if SPI is reused by another team.

* Appendix C: Glossary

- *Tenant*: an isolated customer organization in the platform; 1:1 with a Keycloak realm.
- *Realm*: a Keycloak namespace containing users, clients, roles, groups.
- *Client*: a Keycloak OAuth2 client; represents a deployable application.
- *Resource*: a protected entity in Keycloak's authorization services (e.g., =order=).
- *Scope*: an action verb in Keycloak's authorization services (e.g., =view=).
- *Permission*: a Keycloak authz permission binding (resource + scope + policy).
- *Policy*: a Keycloak authz rule (role-based, time-based, JS-based, etc.).
- *RPT (Requesting Party Token)*: a JWT augmented with UMA-evaluated permissions.
- *UMA (User-Managed Access)*: an OAuth2 profile for authorization; we use the
  =urn:ietf:params:oauth:grant-type:uma-ticket= grant to fetch permissions.
- *JWKS (JSON Web Key Set)*: the realm's public keys for verifying JWT signatures.
- *JWT (JSON Web Token)*: the signed token format Keycloak issues.
- *JTI (JWT ID)*: the unique identifier claim on a JWT; used as our cache key.
- *CRD (Custom Resource Definition)*: a Kubernetes object schema extension.
- *Operator*: a controller that reconciles desired state in CRDs to actual state.
- *SPI (Service Provider Interface)*: Keycloak's extension hook mechanism.
- *Helm subchart*: a chart packaged as a dependency of another chart.
- *Helm umbrella chart*: an application chart whose main role is composing subcharts.
- *PKCE (Proof Key for Code Exchange)*: an OAuth2 extension required for SPA flows.
- *BFF (Backend For Frontend)*: an architecture pattern where a per-frontend
  backend mediates between the SPA and downstream services, holding tokens server-side.
- *External Secrets Operator*: a Kubernetes operator that syncs external secret
  stores into native k8s Secrets.
- *MDC (Mapped Diagnostic Context)*: Logback's per-thread context for structured logs.
- *SLI (Service Level Indicator)*: a measurable indicator of service quality.
- *p95 / p99*: 95th / 99th percentile latencies.

** Per-app cutover runbook

For each app in the order above, perform the following steps. The
example uses =skybot=; substitute hostname, namespace, and service for
other apps.

#+begin_src bash
# 0. Preflight: the new "auth-platform" tunnel and OpenResty service exist.
#    See Phase 0/1 exit criteria.

# 1. Add the app's hostname to OpenResty routing table
ssh beelink1 'kubectl exec -n auth-platform deploy/redis -- \
  redis-cli HSET host:skybot.dhyanraj.dev \
    backend "skybot-nginx.skybot.svc.cluster.local:80" \
    tenant  "skybot" \
    auth    "none"'   # "none" until Keycloak migration

# 2. Add an ingress rule to the unified tunnel in Cloudflare dashboard:
#    Hostname: skybot.dhyanraj.dev   Service: http://openresty.auth-platform.svc.cluster.local:80
#    Save. Do NOT change DNS yet.

# 3. Validate via the tunnel UUID hostname directly (bypassing DNS):
curl -H 'Host: skybot.dhyanraj.dev' https://<new-tunnel-uuid>.cfargotunnel.com/health

# 4. Cut DNS: in Cloudflare DNS, change CNAME skybot.dhyanraj.dev
#    from <old-tunnel-uuid>.cfargotunnel.com to <new-tunnel-uuid>.cfargotunnel.com.
#    TTL was likely 300s; wait ~10min for propagation.

# 5. Validate end-user URL works:
curl -v https://skybot.dhyanraj.dev/health

# 6. Monitor for 24h: cloudflared logs on beelink1 for the OLD tunnel
#    should show traffic dropping to zero.
ssh beelink1 journalctl -u cloudflared-skybot.service -f --since '1 hour ago'

# 7. Decommission the old tunnel:
ssh beelink1 'sudo systemctl disable --now cloudflared-skybot.service && \
              sudo rm /etc/systemd/system/cloudflared-skybot.service && \
              sudo systemctl daemon-reload'

# 8. Delete the corresponding Tunnel object in Cloudflare dashboard.
#+end_src

** Rollback procedure

Until step 7 (systemd unit removal) the rollback is a single DNS change:
- Re-point the CNAME from the new tunnel UUID back to the old tunnel
  UUID.
- TTL was 300s --- service is restored in 5 minutes or less.
- Both tunnels keep running side-by-side throughout the cutover window;
  the old one carries zero traffic post-cutover but stays alive until you
  explicitly disable it.

After step 7, rollback requires recreating the systemd unit and the
Cloudflare Tunnel object --- much more expensive and error-prone. Do not
do step 7 until the old tunnel has sat idle for at least 24 hours.

** Cloudflare dashboard cleanup

Post-cutover, the Cloudflare dashboard state should be:
- =Zero Trust -> Networks -> Tunnels=: exactly one tunnel named
  =auth-platform= (or similar).
- All other tunnels (=safe-sound-homes=, =ai-debate=, =dropwatch=, =maya=,
  =skybot=, =wisefolio=, =aicareers=) deleted.
- =DNS=: every previously per-tunnel CNAME now points at the single new
  tunnel UUID.

~NOTE:~ Cloudflare bills per request, not per tunnel. The consolidation
is for operational sanity, not cost reduction.

** High availability for the unified tunnel

A single-tunnel single-host deployment is a *regression* in availability
compared to the current 7-on-1 layout (which is already a SPOF on
=beelink1=, but with seven independent failure domains). Mitigations, in
preference order:

1. Install the unified =cloudflared= as a systemd unit on *two* beelinks
   (e.g., =beelink1= and =beelink2=), both using the same tunnel token
   (=--token=). Cloudflare load-balances across all connected replicas
   of a tunnel and seamlessly drains traffic during a disconnect.
2. Run =cloudflared= as a Deployment *in* the k3s cluster (image
   =cloudflare/cloudflared=, =replicas: 2=, pod anti-affinity across
   nodes). Token stored as a Kubernetes Secret. This eliminates the
   host-systemd dependency entirely.
3. Recommend (2) for the long term --- it keeps the tunnel's lifecycle
   on the same control plane as everything else.

~WHY:~ Running =cloudflared= in-cluster trades a tiny amount of "what if
k3s itself is down" risk for substantial operational gains: declarative
deployment, easy rolling updates, replica count tunable like any other
workload, secrets managed alongside everything else. The off-cluster
systemd pattern made sense when there was no cluster to host it on;
that's no longer true.
