** [#A] #115 — =cf-setup-tunnel.sh= wipes dynamically-added tenant ingress rules
   :PROPERTIES:
   :TASK: 115
   :STATUS: code-complete; deploy-pending

** [#A] #92 — CNPG superuser password drift on every redeploy
   :PROPERTIES:
   :TASK: 92
   :STATUS: code-complete; migration-pending
   :END:

   CNPG regenerates the =postgres= superuser secret on every helm upgrade
   that touches the cluster CR (or sometimes for no obvious reason at all).
   Our auth-manager pods read the superuser password from
   =platform-pg-superuser-secret= at boot — if the secret was rotated but
   the PG cluster still has the *previous* password, auth-manager 401s out
   on its bootstrap connection.

   Root cause: neither CR set =enableSuperuserAccess=, which defaults to
   *false* since CNPG 1.21 — so CNPG disables the =postgres= password and
   re-asserts that on every reconcile.

   *Code fix landed (approach (a)):* =enableSuperuserAccess: true= +
   operator-owned, pinned basic-auth superuser secret (=username=postgres=).
   With a user-supplied superuser secret, CNPG syncs the role password TO
   the secret on each reconcile (no rotation), eliminating the drift.
   - Prod CR =deploy/cnpg/pg-cluster.yaml=: flag added; header documents the
     operator pre-creating the basic-auth secret.
   - Dev =scripts/deploy-beelink-dev.sh=: now pre-creates
     =platform-pg-dev-superuser-secret= (basic-auth, =PG_SUPERUSER_PASSWORD=
     env var) and sets the flag on the inline CR.
   - Docs updated (OPERATOR-GUIDE: #92 entry, rotation, recovery runbook,
     deploy pre-steps marked LEGACY; one-time migration note added).

   *Remaining work — one-time live-cluster migration:* the EXISTING prod +
   dev clusters were created without the flag + with CNPG-generated secrets,
   so they still drift until migrated. Steps: create the basic-auth
   superuser secret with the password the DB currently uses, ensure the CR
   has =enableSuperuserAccess: true= (re-apply =pg-cluster.yaml= for prod),
   reconcile. See OPERATOR-GUIDE "CNPG superuser pinning — one-time
   migration". Did NOT touch =CnpgSuperuserDataSourceConfig.java= /
   =TenantDatabaseService.java= — the app already reads the secret correctly.

** [#B] #94 — Wizard / IdP setup: auto-link IdP login to operator-provisioned local user
   :PROPERTIES:
   :TASK: 94
   :STATUS: pending
   :END:

   When an operator provisions =dhyan@example.com= as the admin user and
   enables Google IdP, then logs in via Google with the same email — KC
   creates a *brokered* user with a different UUID instead of linking to
   the local account. Operator ends up with two accounts: the local one
   has the roles, the brokered one is empty → 403 on everything.

   *Fix:* enable KC's =Automatic Link= first-broker-login authenticator
   (variant of the existing First Broker Login flow) for the platform +
   tenant realms during creation. Match on email. If a local user with
   that email exists, link the IdP identity to it instead of creating a
   new user.

   Realm-creation flow lives in =KeycloakAdminService.createRealm= and
   =IdentityProvidersBootstrap= for platform realm. Both need the new
   first-broker-login flow wired.

** [#D] BFF single-flight refresh + KC reuse-detection policy review
   :PROPERTIES:
   :TASK: (not filed — was P1 before #118 shipped, now P3)
   :STATUS: pending
   :END:

   Maya claude's hypothesis was that concurrent /api/* bursts could
   cause two parallel refresh attempts → first rotates the refresh
   token, second submits the stale one → KC refuse-detection revokes
   the token family → session dies. Evidence didn't actually support
   this (the failure mode we hit was cookie Max-Age expiry, not refresh
   reuse — Redis session was alive when "death" occurred), but it's a
   latent issue under burst load.

   *If we ever decide to fix:*
   - Add a Redis-based mutex in =maybe_refresh= (=SET NX PX=) so only
     the first concurrent request actually calls KC; the others wait
     and re-read the session
   - OR set realm's =refreshTokenMaxReuse=1= to give us one free retry
     under race

   Currently P3 because the symptom is gone and we have no evidence of
   the race actually firing.

** [#D] BFF dead-cookie silent 401 loop UX
   :PROPERTIES:
   :TASK: (not filed)
   :STATUS: pending
   :END:

   If the browser has a =bff_sid= cookie but the Redis session is gone
   (TTL'd out while tab was backgrounded, manual session invalidation
   from admin, etc.), every /api/* returns 401 with no =Set-Cookie:
   bff_sid=; Max-Age=0= to clear the dead cookie. SPAs either need to
   redirect-on-401 (Maya's wrapper) or sit there showing errors.

   Cleaner contract: when load_session returns nil for a sid that the
   browser sent, BFF should =clear_session_cookies()= so the next
   request goes through the no-cookie path → BFF auto-redirects to KC.
   Already done in some paths (=/_bff/me= line 730, =maybe_refresh=
   failure paths) but not in the catch-all =enforce_auth= →
   =try_inject_bearer_from_cookie= path.

   ~5-line fix in bff.lua but worth verifying we're not breaking any
   intentional silent-401 callers first.
