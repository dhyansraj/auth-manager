** [#A] #92 — CNPG superuser password drift on every redeploy
   :PROPERTIES:
   :TASK: 92
   :STATUS: DONE — enableSuperuserAccess + pinned basic-auth superuser secret;
            applied to prod live.
   :END:

   CNPG regenerates the =postgres= superuser secret on every helm upgrade
   that touches the cluster CR (or sometimes for no obvious reason at all).
   Our auth-manager pods read the superuser password from
   =platform-pg-superuser-secret= at boot — if the secret was rotated but
   the PG cluster still has the *previous* password, auth-manager 401s out
   on its bootstrap connection.

   Root cause: neither CR set =enableSuperuserAccess=, which defaults to
   *false* since CNPG 1.21 — so CNPG disables the =postgres= password and
   re-asserts that on every reconcile.

   *Code fix landed (approach (a)):* =enableSuperuserAccess: true= +
   operator-owned, pinned basic-auth superuser secret (=username=postgres=).
   With a user-supplied superuser secret, CNPG syncs the role password TO
   the secret on each reconcile (no rotation), eliminating the drift.
   - Prod CR =deploy/cnpg/pg-cluster.yaml=: flag added; header documents the
     operator pre-creating the basic-auth secret.
   - Dev =scripts/deploy-beelink-dev.sh=: now pre-creates
     =platform-pg-dev-superuser-secret= (basic-auth, =PG_SUPERUSER_PASSWORD=
     env var) and sets the flag on the inline CR.
   - Docs updated (OPERATOR-GUIDE: #92 entry, rotation, recovery runbook,
     deploy pre-steps marked LEGACY; one-time migration note added).

   *Remaining work — one-time live-cluster migration:* the EXISTING prod +
   dev clusters were created without the flag + with CNPG-generated secrets,
   so they still drift until migrated. Steps: create the basic-auth
   superuser secret with the password the DB currently uses, ensure the CR
   has =enableSuperuserAccess: true= (re-apply =pg-cluster.yaml= for prod),
   reconcile. See OPERATOR-GUIDE "CNPG superuser pinning — one-time
   migration". Did NOT touch =CnpgSuperuserDataSourceConfig.java= /

** [#C] email-byo-sendgrid — bring-your-own SendGrid API key per tenant
   :PROPERTIES:
   :TASK: byo-sendgrid
   :STATUS: pending
   :END:

   Today ALL tenant mail funnels through one platform SendGrid account via a
   single global key (=sendgrid-api= Secret / =SENDGRID_API_KEY=), used in two
   places: the =smtp-relay= (Go, auths to =smtp.sendgrid.net:587= as
   =apikey=/<platform key>) for KC + auth-manager SMTP sends, and
   =SendGridClient= (Web API v3) for per-tenant domain-auth under *our* account.
   Per-tenant today = authenticated domain + From + DKIM, but all under the
   platform account. There is NO per-tenant API key field anywhere (DB, entity,
   settings) — confirmed.

   BYO lets a tenant send as their OWN SendGrid account. The win over the
   current model (which already isolates domain/DKIM/From): the tenant's own
   quota/billing, sender-reputation isolation (their spam complaints don't touch
   other tenants), and their own analytics/suppression lists. Strongest pull is
   a tenant who ALREADY runs SendGrid in prod (e.g. WiseFolio) and wants to keep
   their existing account, templates, and suppression history.

   *Design — BYO tenants route AROUND the relay entirely; platform key stays the
   default for everyone else (zero regression).* The key touches three
   independent send paths, each must switch to the tenant key:
   1. *KC-originated mail* (reset/verify/invite — the bulk). KC realm SMTP
      supports per-realm auth: =SmtpConfigBootstrap= points the BYO realm's SMTP
      straight at =smtp.sendgrid.net:587=, =auth=true=, =starttls=true=,
      user==apikey=, pass=<tenant key> — bypassing the relay. No relay change.
   2. *auth-manager transactional mail* (=EMAIL_SEND= API + invitations). Send
      via =SendGridClient= Web API v3 =mail/send= with the tenant key (reuses
      the existing HTTP client) instead of JavaMailSender→relay.
   3. *Domain auth* — run create/validate against the tenant's account with
      their key (or skip; they manage their own domain). =isTenantAuthenticated
      Domain= flips to "verified in tenant's account."

   *The hard part is the secret, not the plumbing.* The key is a live,
   full-access SendGrid credential. We have NO app-level secret encryption today
   (config sits in plaintext JSONB), so BYO needs net-new:
   - encryption at rest — =settings.sendgrid.apiKeyEnc= via AES-GCM with a master
     key from a platform Secret, OR seal each tenant key into its own k8s Secret
     (do NOT add a plaintext column)
   - validation on entry (test call before save), rotation, revocation, never
     log it, admin-UI to enter/replace
   Send-path edits are ~a day; the encryption + key-management UX is the real
   cost. Not gating anything. Related: [[email-compliance]] (suppression lists
   become tenant-owned under BYO).

** [#B] bff-self-recovery — BFF recovers instead of dead-ending the user (epic)

   ever fixed: Redis mutex in =maybe_refresh= (=SET NX PX=) so only the first
   request calls KC; OR realm =refreshTokenMaxReuse=1= for one free retry.
   Deliberately kept OUT of 0.1.23 — a speculative concurrency change doesn't
   belong in a UX-recovery roll.
